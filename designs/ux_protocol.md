## 検証環境と計測手法（全画面共通）

### 検証環境（3環境）

| ID | 環境 | 入力 | 用途 |
| --- | --- | --- | --- |
| **E1** | Mac Chrome 最新版（`npm run dev`） | マウス | 開発反復・フレーム精度の録画検証。触覚は検証不可 |
| **E2** | iOS 16+ 実機（Capacitor シェル） | タッチ | 触覚（prepare 済み UIImpactFeedbackGenerator 相当）・セーフエリア・縦固定の検証 |
| **E3** | 中級 Android 実機（Snapdragon 6xx / Helio G 系、Android 10+）Capacitor WebView | タッチ | **性能判定の基準環境**（[KPI-001](./README.md#kpi-001): 60fps・物理 step p95 ≤ 4ms）+ areAllPrimitivesSupported() フォールバックの検証（[FR-014](./functional_requirements.md#fr-014)） |

### 計測手法

| ID | 手法 | 手順 | 精度 |
| --- | --- | --- | --- |
| **V-SS** | 静止スクリーンショット | 各画面の指定状態を撮影し、表示要素チェックリストと1項目ずつ照合する | 要素の有無・色・配置 |
| **V-REC** | 60fps 画面録画のフレームカウント | iOS: 標準画面収録 / Android: `adb shell screenrecord` / Mac: QuickTime。録画をコマ送りし、イベント間のフレーム数 × 16.7ms で時間を算定する | ±1フレーム（±16.7ms） |
| **V-CAM** | 外部カメラ 240fps スロー撮影 | 指と画面を同時に写し、タッチ→表示反映の遅延を計測する（入力遅延専用。V-REC はタッチ座標が映らないため） | ±4.2ms |
| **V-OVL** | デバッグオーバーレイ（[SC-010](#sc-010) / [FR-025](./functional_requirements.md#fr-025)） | fps・物理 step 時間 p95・ボディ数を実行中に読む | エンジン内部計測値 |
| **V-AUD** | 聴感 + 録音波形 | 実機スピーカーで SE の発火・停止・ピッチ変化を確認する。ダッキング（dB）は録画の音声トラック波形で確認する | SE 単位 |
| **V-HAP** | 触覚の手持ち確認 | E2 / E3 の実機を手に持ち、イベント発火時の触覚有無と強弱段階を確認する。**E1（Web）とシミュレータでは検証不可** | イベント単位 |

### gatekeeper HG ゲートでの使い方

1. 各画面の「表示要素チェックリスト」を **V-SS** で1項目ずつ照合する（1画面につき V-SS 最低1枚。状態が複数ある画面は状態ごとに1枚）。
2. 「合格基準」の時間契約項目を **V-REC**（[SC-003](#sc-003) / [SC-004](#sc-004) / [SC-005](#sc-005) / [SC-006](#sc-006) は V-REC 必須）で判定する。
3. 触覚項目は E2 と E3 の両実機で **V-HAP** 判定する（片方のみ合格は不合格）。
4. エビデンスは「画面ID + 状態名」で命名して提出する（例: `sc-003_ink-red.png`, `sc-005_five-beats.mp4`）。
5. 1項目でも未達なら当該画面は不合格とし、推測で「実装されているはず」と判定することを禁止する（実測のみが証拠）。

## 画面1: タイトル/ホーム — [SC-001](./functional_requirements.md#sc-001)

> ソースFR: [FR-016](./functional_requirements.md#fr-016), [FR-018](./functional_requirements.md#fr-018) | 根拠: [research/07_decision.md](../research/07_decision.md) §3.5・§5.2
>
> USER GOAL: プレイを開始する / コイン残高を確認する。FIRST ACTION: Play ボタンをタップ（起動 → 2タップ以内でプレイ開始）。

### 表示要素チェックリスト（V-SS）

- [ ] Play ボタン（最大サイズ・親指圏。タッチターゲット ≥ 44pt）
- [ ] コイン残高（[SC-005](#sc-005) / [SC-007](#sc-007) と同一の値・同一の表記 — [FR-018](./functional_requirements.md#fr-018) P1）
- [ ] 設定入口（アイコン、≥ 44pt）
- [ ] ショップ入口（アイコン、≥ 44pt）
- [ ] タイトルロゴ（名称に競合名 "Draw Bridge" を含まない）
- [ ] RV ボタンその他の広告 UI 導線が**表示されていない**（v1.0 フラグ = 非表示、[FR-022](./functional_requirements.md#fr-022)）

### 操作と期待フィードバック

| 操作 | 視覚 | 音 | 触覚 | 数値契約 |
| --- | --- | --- | --- | --- |
| Play タップ | [SC-002](#sc-002) へ遷移 | タップ音1種 | なし | タッチ→視覚反映 ≤ 100ms（[KPI-002](./README.md#kpi-002)） |
| 設定タップ | [SC-008](#sc-008) へ遷移 | タップ音1種 | なし | 同上 |
| ショップタップ | [SC-007](#sc-007) へ遷移 | タップ音1種 | なし | 同上 |

### 遷移先

| トリガー | 遷移先 | 条件 |
| --- | --- | --- |
| Play | [SC-002](#sc-002) レベル選択 | 常時 |
| 設定入口 | [SC-008](#sc-008) 設定 | 常時 |
| ショップ入口 | [SC-007](#sc-007) ショップ | 常時 |

### 検証手順

1. [E2/E3] アプリをタスクキルしてから起動し、V-REC でアイコンタップ→ホーム操作可能まで ≤ 3秒を確認する（[NFR-006](./non_functional_requirements.md#nfr-006)）。
2. [E1] `npm run dev` の URL を新規タブで開き、初回ロード ≤ 5秒（4G 相当のスロットリング）を確認する（[NFR-006](./non_functional_requirements.md#nfr-006)）。
3. [E2/E3] Play → 解放済みレベルタップの2タップで [SC-003](#sc-003) に到達することを V-REC で確認する（タップ回数を数える）。
4. V-SS: ホームのスクリーンショットを撮り、表示要素チェックリスト6項目を照合する。コイン残高値を控え、[SC-007](#sc-007) / [SC-005](#sc-005) の表示値と一致を確認する。
5. [E2] ノッチ端末でセーフエリア内に全 UI が収まること、横向きにしても縦固定が維持されることを確認する（[FR-023](./functional_requirements.md#fr-023)）。
6. 進行データ破損時（手動でストレージ値を壊す）: 「進行データを復元できなかった」通知が出て初期状態で起動し、クラッシュしないことを確認する（[FR-021](./functional_requirements.md#fr-021) 例外フロー②）。

### 合格基準（HG 判定）

- [ ] コールドスタート ≤ 3秒（実機）/ Web 初回ロード ≤ 5秒
- [ ] 起動 → 2タップでプレイ開始（[SC-003](#sc-003) 到達）
- [ ] 残高が全画面（ホーム / ショップ / リザルト）で同一値
- [ ] 広告 UI 導線ゼロ・外部ネットワーク通信ゼロ（[NFR-012](./non_functional_requirements.md#nfr-012)、DevTools の Network タブで確認）
- [ ] データ破損時に無言初期化しない（通知表示）

TRUST: P1 残高の一貫表示 / P3 破損時の通知付き復元。

## 画面2: レベル選択 — [SC-002](./functional_requirements.md#sc-002)

> ソースFR: [FR-015](./functional_requirements.md#fr-015), [FR-016](./functional_requirements.md#fr-016) | 根拠: [research/07_decision.md](../research/07_decision.md) §3.3（信頼設計 P1 / P6）
>
> USER GOAL: 次に挑む面を選ぶ / 星の埋まり具合を眺める。FIRST ACTION: 未クリアの最新面（強調表示）をタップ。

### 表示要素チェックリスト（V-SS）

- [ ] Ch1 の15面 + ボーナス3面（L5 / L10 / L15 の後）のカードが全て表示されている
- [ ] 各カードに獲得星（0〜3）・クリア状態・ロック状態が表示されている
- [ ] ボーナス面が通常面と視覚的に区別されている（[game_design.md](./game_design.md#gd-7-4) §7.4）
- [ ] 未クリアの最新面が強調表示されている
- [ ] ホームへ戻る導線
- [ ] 各カードのタッチターゲット ≥ 44pt（[NFR-009](./non_functional_requirements.md#nfr-009)）

### 操作と期待フィードバック

| 操作 | 視覚 | 音 | 触覚 | 数値契約 |
| --- | --- | --- | --- | --- |
| 解放済み面タップ | [SC-003](#sc-003) へ遷移 | タップ音 | なし | レベル遷移 ≤ 1秒（[NFR-006](./non_functional_requirements.md#nfr-006)） |
| ロック面タップ | ロックアイコンのシェイク + 「前のレベルをクリア」表示。遷移しない | なし（無音で拒否しない場合は拒否音1種に統一） | なし | — |
| 戻る | [SC-001](#sc-001) へ遷移 | タップ音 | なし | — |

### 遷移先

| トリガー | 遷移先 | 条件 |
| --- | --- | --- |
| 解放済み面タップ | [SC-003](#sc-003) 描画フェーズ | 当該面が解放済み |
| ロック面タップ | 遷移なし（シェイク + 解放条件表示） | 当該面が未解放 |
| 戻る | [SC-001](#sc-001) | 常時 |

### 検証手順

1. V-SS: 初回起動状態（L1 のみ解放）・中間進行状態（例: L7 まで）・全クリア状態の3状態でスクリーンショットを撮り、星 / ロック / ボーナス区別を照合する。
2. [E1] クリア済み面を星の低い結果で再プレイし、レベル選択に戻って表示星がベスト値のままであることを確認する（[FR-016](./functional_requirements.md#fr-016) 主フロー4）。
3. ロック面をタップし、シェイクと解放条件表示が出て遷移しないことを V-REC で確認する。
4. [E2/E3] 面タップ → [SC-003](#sc-003) の静止俯瞰表示まで V-REC で ≤ 60フレーム（1秒）を確認する。
5. アプリを強制終了 → 再起動し、直近クリアまでの進行が復元されていることを確認する（[NFR-007](./non_functional_requirements.md#nfr-007)）。

### 合格基準（HG 判定）

- [ ] 15面 + ボーナス3面が全カード表示され、ロード待ち（スピナー）が存在しない（ローカル即時）
- [ ] 星のベスト値保持（低い結果で退行しない）
- [ ] ロック面は遷移せず解放条件をフィードバックする
- [ ] 面タップ→プレイ可能まで ≤ 1秒
- [ ] 強制終了→再起動で進行復元

TRUST: P1 星・クリア状態の常時可視 / P6 クリア履歴の一覧性。

## 画面3: ゲームプレイ（描画フェーズ） — [SC-003](./functional_requirements.md#sc-003)

> ソースFR: [FR-001](./functional_requirements.md#fr-001), [FR-002](./functional_requirements.md#fr-002), [FR-003](./functional_requirements.md#fr-003), [FR-004](./functional_requirements.md#fr-004), [FR-010](./functional_requirements.md#fr-010), [FR-017](./functional_requirements.md#fr-017) | 根拠: [research/07_decision.md](../research/07_decision.md) §4.1・§7.2 / [game_design.md](./game_design.md#gd-4-1) §4.1
>
> USER GOAL: 地形を読み、1ストロークで橋を描く。FIRST ACTION: 谷をまたぐ位置をドラッグ。**脳汁3場面の1つ目 — 本画面の検証は juice 必須項目（[KPI-005](./README.md#kpi-005)）の合否に直結する。**

### 表示要素チェックリスト（V-SS）

- [ ] インク残量バー（満量から開始。> 50% 緑 / 20〜50% 黄 / < 20% 赤 + 300ms 周期点滅）
- [ ] レベル番号
- [ ] リスタートボタン（常設、≥ 44pt）
- [ ] 地形 + 車（スポーン位置に静止）+ ゴール旗が1画面の静止俯瞰に収まり、解くべき地形が読める
- [ ] L1 のみ: 指アイコンのなぞり誘導アニメーション（谷をまたぐ軌跡）。**テキストチュートリアルが存在しない**（[FR-017](./functional_requirements.md#fr-017)）
- [ ] 描いた線: 太さ画面幅 2〜3%（375pt 幅で 8〜12pt）・丸キャップ・高コントラスト単色 + 外周 1〜2px 濃色ボーダー

### 操作と期待フィードバック

| 操作 / イベント | 視覚 | 音 | 触覚 | 数値契約 |
| --- | --- | --- | --- | --- |
| タッチダウン→ドラッグ（描画） | 線先端が生タッチ座標を**同フレーム**追従。平滑化は過去点のみ。ペン先ダスト 2〜5個/フレーム（推奨） | ループ音フェードイン 30〜50ms。速度→音量 0.3〜1.0 / ピッチ 1.0〜1.2。ストローク毎に基準ピッチ ±5% | （推奨・任意）continuous | 先端反映 ≤ 16.7ms（同フレーム）、タッチ→視覚 ≤ 100ms（[KPI-002](./README.md#kpi-002)） |
| 指の停止 | 線が伸びない | ループ音停止（フェードアウト 30〜50ms） | なし | 「動き = 音」の一致 |
| インク消費 | バーが同フレームで減少。50% / 20% で色境界 | なし | なし | 描画長比例 |
| インク枯渇 | 頂点追加停止（線が伸びない）+ バー横シェイク 4〜6px / 150ms | 「カスッ」空振り音 | warning（[FR-014](./functional_requirements.md#fr-014)） | 枯渇フィードバック3チャネル同時 |
| 指離し（確定） | 確定ポップ scale 1.0→1.06→1.0 / 120ms（ease-out-back）+ 線がばね物理でわずかに落下・たわむ | 確定音「コトッ」50〜120ms | light / PRIMITIVE_TICK(0.6) | 確定と同時に溜め開始（0秒フィードバック、[FR-003](./functional_requirements.md#fr-003) 主フロー6） |
| 最小長未満で指離し | 描線破棄 + インク返還、描画フェーズに留まる | — | — | 描線 < 0.5m 相当（[FR-003](./functional_requirements.md#fr-003) 例外フロー①） |
| リスタートタップ | 暗転最小で初期状態へ。確認ダイアログを出さない | タップ音 | なし | タップ→復帰 ≤ 1秒（[KPI-003](./README.md#kpi-003)） |

EDGE: OS のタッチ中断（着信・通知シェード・システムジェスチャ）は指離しと同一に扱い確定処理へ進む（[FR-001](./functional_requirements.md#fr-001) 例外フロー②）。E1 のマウス入力はタッチと同一挙動（[NFR-005](./non_functional_requirements.md#nfr-005)）。初回タッチ時に AudioContext が suspended なら resume を先行し、描画をブロックしない（[FR-010](./functional_requirements.md#fr-010) 例外フロー）。

### 遷移先

| トリガー | 遷移先 | 条件 |
| --- | --- | --- |
| 指離し（頂点2点以上・0.5m 相当以上） | [SC-004](#sc-004) 走行フェーズ（描画 UI 非表示化） | 確定成功 |
| リスタート | [SC-003](#sc-003) 初期状態 | 常時（≤ 1秒） |

### 検証手順（秒数・フレーム数付き）

1. **先端追従 [E1]**: L1 を開き、低速（約 100px/s）と高速（約 2,000px/s）の2速度で線を描き V-REC(60fps) で録画する。コマ送りで全フレームを確認し、カーソル位置と線先端のズレが 0 フレームであること（先端が過去点平滑化で遅れないこと）を確認する。
2. **入力遅延 [E2/E3]**: V-CAM（240fps）で指と画面を同時撮影し、タッチ移動→線反映のフレーム差を計測する。合格: ≤ 100ms（240fps で ≤ 24 コマ）、先端は次の描画フレーム（60fps 換算 1 フレーム）以内。
3. **描画音 [E2/E3]**: V-AUD で (a) 描画開始から 30〜50ms 以内に音開始 (b) 指を止めて 30〜50ms フェードで音停止 (c) 描く速度を上げると音量・ピッチが上がる (d) 2ストローク（リスタートを挟む）で基準ピッチが変わる、の4点を確認する。
4. **インクバー3状態 [E1]**: 残量約 60% / 35% / 10% の3時点で V-SS を撮り、緑 / 黄 / 赤を照合する。赤帯の点滅は V-REC で 18 フレーム（300ms）周期を確認する。
5. **枯渇 [E2/E3]**: インクを使い切るまで描き、(a) 線が伸びない (b) 「カスッ」音（V-AUD）(c) バー横シェイク 150ms = 9 フレーム（V-REC）(d) warning 触覚（V-HAP）の4点同時発火を確認する。枯渇後も指を離せば確定に進めることを確認する。
6. **確定→発進の 0秒フィードバック [E1]**: V-REC で指離しフレームを 0 とし、(a) 確定ポップが 7 フレーム（120ms）で完了 (b) 線が落下・たわみ始める (c) 同時に車の溜め（レブ音・後傾）が開始し、無応答フレームが 0 であることを確認する。
7. **確定触覚 [E2/E3]**: V-HAP で light（iOS）/ TICK 相当（Android）を確認する。
8. **リスタート**: 描画途中でリスタートをタップし、V-REC でタップ→初期状態の静止俯瞰まで ≤ 60 フレーム（1秒）、確認ダイアログが出ないことを確認する。
9. **FTUE [E2/E3]**: 進行リセット後に L1 を開き、(a) 指アイコンなぞり誘導が表示される (b) テキストが 0 件 (c) 誘導と違う線でもクリアできる (d) L1 失敗→リトライで誘導が再表示される、を確認する。
10. **性能 [E3]**: V-OVL を表示したまま最長の線（インク満量消費）を描き、描画中 60fps 維持を確認する。

### 合格基準（HG 判定）

- [ ] 線先端が生タッチ座標を同フレーム反映（過去点のみ平滑化）
- [ ] タッチ→視覚 ≤ 100ms・タッチ→触覚 ≤ 100ms（[NFR-002](./non_functional_requirements.md#nfr-002)）
- [ ] 描画ループ音が速度連動し、停止で 30〜50ms フェード停止する
- [ ] インクバーが緑 / 黄 / 赤 + 300ms 点滅の3状態を持ち、枯渇時に音 + シェイク + 触覚の3チャネルが同時発火する

### 検証手順（秒数・フレーム数付き）

1. **溜めの長さ [E1]**: V-REC で確定（指離し）フレームを 0 とし、モーター駆動開始（車輪が前進を始める）フレームまで 18〜30 フレーム（0.3〜0.5s）であることを計測する。溜め中に (a) レブ音ピッチ上昇（V-AUD）(b) 後傾 squash (c) 車輪空転 + 煙、が全て出ること、溜め中に画面をタップしてもスキップされないことを確認する。
2. **解放 [E2/E3]**: V-REC でモーター駆動開始フレームに (a) ダスト 10〜20個 (b) 前方 stretch → 6 フレームで復帰 (c) カメラキック逆方向 8〜16px → 18 フレームで復帰、V-AUD で低域バースト音、V-HAP で medium / THUD(0.8) を確認する。
3. **カメラ追従**: 走行中の V-REC で、車が画面中心より進行方向の逆側に置かれ、進行方向に車体 1〜2 台分の余白（look-ahead）が確保されていることを確認する。
4. **エンジン音 [E2/E3]**: V-AUD で速度上昇に伴うピッチ 1.0〜1.5 の連続上昇 + 0.25 刻みのギア段付きを確認する。クリア / 失敗確定でエンジン音が停止することを確認する。
5. **軋み→破断の順序保証**: [SC-009](#sc-009) エディタ（または長スパン面）で、車の荷重で破断する細長い橋を描いて車を通す。V-REC のコマ送りで (a) 破断フレームより前に必ず 黄→赤 の色変化と軋み音・粉パーティクルが観測できる（前兆なし破断 = 不合格）(b) 破断フレームでクラック音 + 破片 + シェイク + 折れ口ハイライト (c) 孤立チェーン片が数秒後にフェードアウトする、を確認する。V-HAP で軋み帯の弱連打を確認する。
6. **コインコンボ [E2/E3]**: コインがアーチ状（0.1〜0.2s 間隔で取れる配置）の面で連続取得し、V-AUD で取得音が半音ずつ上昇し +12 semitone で頭打ちになること、取得を 1.5s 以上止めた次の取得音が基準ピッチへ戻ることを確認する。
7. **性能 [E3]**: V-OVL を表示し、最大構成（描線 32 セグメント + 車 + 崩落中）で 60fps・物理 step p95 ≤ 4ms を確認する（[KPI-001](./README.md#kpi-001) 判定）。
8. **走行中リスタート**: 崩落の最中にリスタートをタップし、V-REC で ≤ 60 フレーム（1秒）で [SC-003](#sc-003) 初期状態（コイン初期配置含む）に戻ることを確認する。
9. **物理の決定性（実機許容帯）[E1]**: 同一レベルで同一のゴースト解描線を再生し、クリア / 失敗の一致 + 最終車両位置 ε = 0.05m + tick ±30 に収まることを確認する（[NFR-004](./non_functional_requirements.md#nfr-004)）。
10. **120Hz 端末（保有時）**: 120Hz 表示でも同一描線の走行結果が 60Hz 端末と許容帯内で一致することを確認する（物理 60Hz 固定の検証）。

### 合格基準（HG 判定）

- [ ] 溜め 0.3〜0.5s 固定（V-REC 実測 18〜30 フレーム）・スキップ不可・3演出（レブ音 / squash / 空転煙）同時
- [ ] 解放の4点セット（ダスト / stretch / バースト音 / medium 触覚）+ カメラキックが発火する
- [ ] 走行中の操作 UI がリスタートのみ（運転・ブースト操作が存在しない）
- [ ] **破断の前に必ず軋み（色 + 音 + 粉 + 触覚）が観測できる**（応力の二重符号化、[NFR-009](./non_functional_requirements.md#nfr-009)）

- [ ] コイン半音上昇コンボ（上限 +12・1〜1.5s リセット）が機能する

TRUST: P1 応力可視化（白→黄→赤） / P2 走行中もリスタート常設 / P4 イベント種別毎に統一されたフィードバック。
