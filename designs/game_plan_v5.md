## 2. F1 — 岩・危険帯 接触＝即ゲームオーバー（fail 原因・judge 順序・ゲート簡素化）

### 2.1 fail 原因の追加と判定順序（judge ordering）

round-7 で fail 原因に **`hazardContact`** を新設し、**最優先**にする。判定順序（1 tick ごと、上から評価）：

| 優先 | 原因コード | トリガ | 実装契約（実装フェーズ向け） |
|---|---|---|---|
| **1** | **`hazardContact`**（新） | VEHICLE カテゴリが ROCK / DANGERZONE(spike含む) カテゴリと Box2D 接触した最初の tick | 接触イベントで `pendingFail = 'hazardContact'` を latch し、その tick で run を終了。**接触＝即負け**（減速・跳ね返しでなく即死）。UL に fail 原因 `HazardContact` を追加（conventions.md に先に登録・learnings T2）|
| 2 | `tipOver` | chassis 角度 > `tipOverAngleRad`(2.1) | 既存 |
| 3 | `killY`（降格） | `refPoint.y < killY`（`killY = frame.minY-1`、可視窓外） | 既存だが**最低優先・非可視**（§4）。設計面ではほぼ発火しない |
| 4 | `timeout` | tick > `maxTicks` | 既存 |

**なぜ `hazardContact` を最優先にするか**: 岩が車を穴へ突き落とす場合、原因を "落下(killY)" でなく "岩接触" と報告したい。接触は落下に**先行**するので、接触 tick で latch すれば正しい原因が付く（learnings T2 の「event tick を機械固定」に整合）。失敗オーバーレイの文言・hazard 帰属もこの原因コードに基づく。

**接触判定の実装契約（レビュー HIGH 反映・曖昧さ排除）**:
- **検出点**: Box2D の **BeginContact**（`b2Contact` の begin-touch イベント）で判定。PreSolve での減速・反発は行わない（接触＝即死のため solve 前に latch し、その fixed-step 終端で run を `Fail(hazardContact)` に確定）。
- **対象 fixture**: **VEHICLE カテゴリの全 fixture（chassis + 両 wheel）** のいずれかが ROCK / DANGERZONE カテゴリと begin-touch した時のみ。橋(BRIDGE)や地形と hazard の接触は無視（車以外の多点接触は fail にしない）。
- **spawn 直後の猶予**: **なし**。ただし level JSON で岩を車と**初期重なりで spawn しない**ことを規約化（authoring 契約：t=0 で車 AABB と岩円が非交差）。生成直後の重なりによる誤 fail を設計側で排除（gate で t=0 非交差を検証）。
- **同 tick で goal 到達と hazard 接触が競合**した場合: **hazard 接触を優先**（＝clear より fail が勝つ）。ゴール線上に危険を置かない authoring 規約と併せ、判定順序を決定論的に固定（learnings T2）。
- 上記の順序・対象・優先を contract テストで機械固定する。

### 2.2 危険は「走行筋道の上」に置く（避けさせる設計）

- v4 までは危険を「pit＋killY（落ちたら失格）」で表現していたが、round-7 では **危険体（岩・スパイク・危険帯）を車の素直な走行ライン上に配置**し、**正しいストロークで回避／盾／逸らし**をさせる。
- アトラス各カードに **naive path（赤破線＝素直な道が危険へ直進）＋ ✗（接触即死点）＋ 正解 carPath（緑破線＝回避）** を明示（[atlas-design-v5.html](./atlas-design-v5.html)）。これが「危険を走行筋道上に置いて避けさせる」の図的定義。

### 2.3 hazard-relevance ゲートの簡素化（contact IS loss）

**v4 の課題**（learnings §A2 T4）: relevance ゲートは岩が「dominance を壊す」ことを間接的に証明していたため、rim-to-rim の空虚な合格が起きた。

**v5 の簡素化**: 接触＝損失なので、ゲートは以下の**ブール接触判定**に還元される：

```
各 hazard-tag 面について:
  (正の主張) naive/StraightLineBot ストローク → 実行 →
      VEHICLE が hazard 体に接触する（contactCount ≥ 1）ことを assert
      かつ run が cause='hazardContact' で fail することを assert
  (負の対照) ghost 正解ストローク → 実行 →
      VEHICLE の hazard 接触が 0 で clear することを assert
```

- **正の主張** = 危険は本当に致命的（＝機械的に relevant）。
- **負の対照** = 正解は危険を避けられる（learnings T4「pass する検証に必ず fail ケースを添える」を恒久化）。
- v4 の「anti-dominance bot が overlap {0,1,2}m で落下」複雑判定は不要になり、**接触の有無**という単純不変量に置換される。overlap 校正は「naive パスが確実に接触する初期条件」を spike-sweep で1回確定（learnings T3・ハンドチューニング禁止）。

### 3.2 portrait の制約と3つの規模指標（3指標すべてを gate する）

縦画面では**横は希少資源・縦は潤沢**（研究09 §3.3）。可読ズームの都合で**水平フットプリントは概ね ≤18m**（390px 幅で 1.44m 車が ~31px。それ以上は車が小さく潰れる）。→ 直線 D_sg の**理論上限は可読窓対角に律速され ~14-15m**（横18m + 余白）。よって "近さ/狭さ" は3指標で定義し、**3指標すべてを難度帯下限で gate する**（どれか1つでは体感を取り逃す、というレビュー指摘を反映）：

| 指標 | 定義 | ユーザー不満との対応 | ≥14/≥18 例示の割当 |
|---|---|---|---|
| **可読窓 W_win**（対角） | frame の対角長 = √(W²+H²) | 「ステージ面積が狭い」＝面積 | **「playable window 16-26m」に一致**（XL ≥22、最大24.8）|
| **コース長 L_path** | spawn→ゴールの車の**実走行距離** | 「車とゴールが近すぎ」の体感＝**走りが短い** | 「≥18 終盤」の "長い走り"（L19 が 19.3m、XL 平均 15.6m）|
| **spawn→ゴール分離 D_sg** | spawn と goal 中心の直線距離 | 直線的な "遠さ" | portrait 可読上限内で漸増（XL ≥12m、最大14.3m）|

> **設計判断（透明性のため明記・レビュー反映）**: タスク例示「≥14m 中盤 / ≥18m 終盤」を **spawn→ゴール直線 D_sg** にそのまま適用すると portrait 可読上限（水平≤18m → D_sg 上限 ~14-15m）と衝突する（直線18m＝横17m級で車が潰れる）。よって ≥14/≥18 は **可読窓（＝タスクが対で挙げる「playable window 16-26m」）と コース長（＝"長い走り"）に割り当て**（両方 XL で達成）、直線 D_sg は可読上限内で**中盤≥10 / 発展≥11 / 頂点≥12m** まで漸増させる（現状 median 8.5m からの明確な増分・全面が下限クリア）。felt closeness の主因は走行距離だが（研究09）、D_sg も見捨てず gate 対象に含める（レビュー HIGH 反映）。

## 4. F3 — killY の設計的廃止

| 項目 | v4 まで | v5（round-7） |
|---|---|---|
| level JSON の `killY` | 設計値（谷底の失格ライン） | **out-of-world engine failsafe のみ**。`killY = frame.minY - 6`（可視窓の遥か下）に据え、通常プレイでは到達しない |
| ゲーム内描画 | 破線＋"killY" ラベル | **描画禁止**（実装フェーズで RenderScene から killY 描画を除去する契約）|
| アトラス描画 | 破線＋ラベル | **除去済**（atlas-design-v5.html は killY を一切描かない）|
| 設計要素としての扱い | 「落ちたら失格」を危険として使用 | **廃止**。落下＝自明の失敗。危険は in-path のみ |
| ユーザー向け fail 原因 | `killY`（谷落ち）を表示 | **表示しない**。落下は `tipOver`（着地失敗）または（万一の脱出時のみ）内部 failsafe に集約。失敗オーバーレイに "killY/落下ライン" の語を出さない |
| judge 優先度 | 通常 fail | **最低優先・非可視**（§2.1）。良設計面ではほぼ発火しない |

- **根拠（レビュー MEDIUM 反映）**: 「下に落ちたら失格」は自明であり設計変数として無意味（ユーザー指摘 R7-3）。`minY-1` だと可視窓直下で「見えない線で突然 fail」に見えるため、**failsafe を frame.minY-6 まで下げ**、通常の谷落ちは着地転倒(`tipOver`)や走行不能で先に決着させる。killY はワールド外脱出という**異常系専用**にし、ユーザー向け原因コードには出さない。
- **failsafe を残す理由**: 車が万一ワールド外へ出ても sim を確定終了させ、決定論とテンポを守る（設計上は見えず、内部専用）。
- **UL/conventions**: `killY` の意味を「設計 danger」から「out-of-world engine failsafe（非表示・異常系専用）」に**再定義**（両方向整合・learnings T2）。禁止語彙は現状維持。

## 5. F4 — 28面スレート（新ラインナップ）

### 5.1 構成の選択と競合チャプター規模との比較

- **23 番号面（L1-L23）＋ 5 ボーナス（B1-B5）＝ 28面**を採用。
- **なぜ 23 番号面**: 9役割タクソノミ（§6）を「導入→定着→複合」で展開するのに ~2.5 枠/役割 必要（9×2.5≈23）。
- **なぜ 5 ボーナス**: 鋸歯の "谷"（息抜き）を **番号面 ~4-5 面ごとに1枚**挿入（研究09 §3.5）。L4→B1 / L7→B2 / L11→B3 / L15→B4 / L23→B5。
- **競合チャプター規模との整合**: Happy Glass の1"world" ≈25面 / Candy Crush 1エピソード ≈15面 / Draw Climber・Bravestars の1 world 20-80面。**28面は Chapter 1 として妥当**（研究09 §3.1、DESIGN.md §2.1）。

### 6.2 検証結果（機械計算・独立検証 VP-1）

- 各面を5軸の署名にエンコードし、全ペア（28×27/2 = **378ペア**）の「異なる軸数（Hamming距離）」を計算（`scratchpad/variety.mjs`）。
- **ルール「no two levels within 2 axes」= 全ペアが Hamming距離 ≥3（＝相違が3軸以上・一致は最大2軸）**（v4 の ≥2 から強化）。
- **結果: MIN 相違 = 3 / FAIL(＜3)ペア = 0 件。** 全378ペアが5軸中3軸以上で相違（＝一致≤2軸）。**"混同" を「相違≤2（一致≥3軸）」と定義すると、その混同ペアは 0 件**（レビュー HIGH：用語の逆転を修正。「≥3軸相違」なら「一致は最大2軸」が正しい含意）。
- **規模軸を除いた頑健性（レビュー HIGH 反映）**: 規模クラス Z は "同じ遊びのサイズ違い" で距離を稼げるため、Z を除いた**プレイ差分4軸（地形/危険/役割/線形）でも独立に検証** → **MIN 相違 = 2 / 相違＜2のペア 0 件**。つまり規模を無視しても全ペアが少なくとも2つのプレイ軸で異なる。多様性は規模水増しに依存していない。
- 隣接±1 で同一役割タグ 0 件（§5.2 の順序で検証）。役割の最短再登場間隔は全て3面差以上でサブ役割違い。
- 軸の値分布（偏りなし）: 地形 flat3/climb4/U5/descent5/tier5/S4/shaft2、危険 none7/rFall3/rRoll5/sFloor3/sCeil2/zone2/comp6、役割 road3/seal3/ramp2/shStat3/shDyn2/catch5/hook3/sag3/dome4、規模 S4/M7/L10/XL7、線形 line2/arch3/uarch3/scurve5/dome4/zig4/cant4/ramp3。

> v4 は「min 2（唯一のフロア L1/L3）」だった。v5 は **road tutorial を L1(flat/S/line)・L2(climb/M/ramp) の2面に絞り、規模クラスを違える**ことで road/none 面同士も距離3を確保し、**例外なく全ペア≥3**（規模除外4軸でも≥2）を達成した。

## 8. Feasibility・エンジン制約順守（実装ゲート）

### 8.1 新/既存エンティティの契約

- **Rock**（既存 RockHazard・conventions 既定）: 円 body、`{x,y,radius,density?,initialVelocity?{x,y}}`。重力落下/転がり、地形・線・車と通常接触。**round-7: 車と接触したら `hazardContact` で即 fail**（§2.1。従来の tipOver/fall 経由に依存しない直接 fail）。
- **DangerZone / Spike**（v4 で追加済）: 静的矩形、`{x,y,w,h,kind}`（zone / spike / spikeDown）。車が接触＝即 fail（`hazardContact`）。
- 新規 fail 原因 `HazardContact` は **conventions.md / UL に先に追加**（CLAUDE.md「New entity → conventions.md first」/ learnings T2）。

### 8.2 既存物理制約の順守

- **1 ストローク/試行**（`commitStroke` が2本目で throw）→ 全28面 1本で解ける形。
