# ギャップ補完調査 06: 描線の物理表現アーキテクチャ — 「たわみ・崩落」と剛体compound推奨の矛盾解消

- 調査日: 2026-07-07
- 調査者: ギャップ補完調査員（Claude）
- 対象ギャップ: 市場調査(01)・juice調査(03)が特定した快感の核「線がたわみ、軋み、崩れる見世物」と、技術調査(05)の推奨「単一rigid compound body（カプセル連結）」の未解決矛盾
- 前提スタック: Phaser 4 + Phaser Box2D（Box2D v3.0 のJSポート）+ Vite + Capacitor

## 1. 結論（TL;DR）

**推奨: 「セグメント化カプセルチェーン + ばね付きジョイント + 破断閾値」を第一候補とし、05の単一剛体compoundは"フォールバック層"に格下げする。**

1. **性能は制約にならない**。描線1本=セグメント10〜40なら「剛体 約45個＋ジョイント 約45本」で、これは Box2D 公式 Driving サンプル（カプセル20枚+revolute jointの橋を車が渡る——本作のコアループそのもの）や Bridge サンプル（160枚）が実証済みの負荷帯。JS系エンジンのベンチでも数千剛体まで60fpsが出ており、中級Android WebViewで5〜10倍の減速を見込んでも2桁の余裕がある（確度: 中〜高、§4）。
2. **本当の制約は「安定性チューニング」**（重い車 × 軽いセグメントの質量比、ジョイント伸び）。これは Box2D v3 の Soft Step solver + sub-stepping（4→8）で対処可能とメンテナ本人が明言しており、まさにv3を選んだ利点が効く領域（確度: 高、§2.2/§4.3）。
3. **ジョイント破断は Box2D v3 の一次サポート機能**。`b2Joint_GetConstraintForce` / `b2Joint_GetConstraintTorque` を毎ステップ読み、閾値超過で `b2DestroyJoint` — 公式 BreakableJoint サンプルがそのまま雛形になり、Phaser Box2D は3関数すべてをエクスポート済み（ソースを直接確認、確度: 高、§3）。
4. **市販クローン（Bravestars/Eureka/Eyestorm/CodeCanyonテンプレート）の線はほぼ確実に「単一剛体」**で、たわみは回転・シーソー・落下の錯覚。つまり「本当に軋んで折れる線」は**このジャンルでまだ誰もやっていない差別化ポイント**である（確度: 中、§5）。
5. 着手週の実機スパイクは「同一シーンで A:単一剛体 / B:自由ヒンジ鎖 / C:ばね付きヒンジ鎖 / D:weld(angularHertz)鎖」を切替比較する設計にする（§6.5）。

## 2. 方式比較: 描線を物理化する4+1方式

### 2.1 比較表

| 方式 | たわみ表現 | 部分破断 | 車両走行の安定性 | 実装コスト | 物理コスト | 総評 |
|---|---|---|---|---|---|---|
| **A. 単一剛体compound**（05の推奨。カプセルfixture×Nを1ボディに） | **不可**（剛体なので原理的に曲がらない）。落下・回転・シーソーの「動き」のみ | **不可**（fixture削除で欠けは作れるが、力学的な折れではない） | **最高**（内部自由度ゼロ。ジッタ無し） | 最小 | 最小（1 body） | 快感の核を捨てる。フォールバック用 |
| **B. 多関節セグメント（revolute joint、ばね無し）** | ロープ的に**垂れる**（曲げ剛性ゼロなので「板のたわみ」ではなく「鎖」になる） | **可**（joint破壊で分断） | 中（車の荷重で谷折りに崩れやすい。limit必須） | 中 | 中（N body + N-1 joint） | Box2D公式 Driving サンプルの橋がこの方式 |
| **C. ばね付きヒンジ鎖（revolute + enableSpring/hertz/dampingRatio + limit）** | **板らしい弾性たわみ**（hertzで曲げ剛性、limitで最大折れ角を制御） | **可**（force/torque閾値で破断） | 良（ばねが復元力になりジッタも減衰） | 中 | 中（Bと同等） | **本命**。Bridge サンプル（hertz=2.0, damping=0.7）が実証 |
| **D. weld joint鎖（linearHertz/angularHertz>0）** | **弾性梁**。Box2D公式ドキュメントが「soft-body simulation を模倣するためのばね」と明記 | **可**（同上） | 良〜高（相対回転自体を拘束するため形状保持が強い） | 中 | 中 | Cの代替。硬めの「一枚板がしなる」表現に最適 |
| E. 疑似ソフトボディ（距離ジョイント格子/粒子+外皮） | 最高（ぐにゃぐにゃ） | 可だが制御困難 | 低（車が沈む・貫通しやすい） | 高 | 高（body/joint数が2〜4倍） | 本作にはオーバーキル。Happy Glass系の液体・ゼリー向け |

- 出典（B/Cの実在実装）: Box2D 公式サンプル [sample_joints.cpp](https://github.com/erincatto/box2d/blob/main/samples/sample_joints.cpp) — `Driving` クラスは「カプセル20枚（各 長さ2m×半径0.125m）を revolute joint で連結した橋」を車（wheel joint×2, hertz=5.0, dampingRatio=0.7, torque=5.0）が渡るシーンで、**本作のコアループと完全に同型**（確度: 高、ソース直接確認）。
- `Bridge` クラスは160枚の板を `enableSpring=true, hertz=2.0, dampingRatio=0.7, enableMotor=true(摩擦トルク200)` の revolute で連結（確度: 高、同上）。
- 出典（Dのばねフィールド）: [b2WeldJointDef](https://docs.rs/box2d_sys/latest/box2d_sys/struct.b2WeldJointDef.html) / [Box2D types.h](https://box2d.org/documentation/types_8h_source.html) — `linearHertz / angularHertz / linearDampingRatio / angularDampingRatio`（0で最大剛性）。「The weld joint constraint provides springs to mimic soft-body simulation」（確度: 高）。
- Phaser Box2D（JSポート）側の対応確認: `b2RevoluteJoint_EnableSpring / SetSpringHertz / SetSpringDampingRatio`、weld def の `linearHertz / angularHertz` をソースでエクスポート確認（[src/include/joint_h.js](https://github.com/phaserjs/phaser-box2d)、確度: 高）。

### 2.2 「たわみ」の要素分解 — 矛盾の正体

01/03 が快感と特定した「たわみ・崩落」は、実際には3つの異なる現象の合成である:

1. **配置のたわみ**（指を離した瞬間、線が重力で落ちて地形に馴染む）— これは方式Aでも起きる（落下・回転）。ただしAでは線は真っ直ぐなまま傾くだけ。
2. **荷重のたわみ**（車が乗ると線が下に凹み、軋む）— **方式B/C/Dでしか起きない**。03が「必須」とした「物理化の視覚合図（確定と同時に線がわずかに落下・たわむ）」はここに属する。
3. **限界の崩落**（凹みが限界を超えて折れ、車ごと落ちる見世物）— **ジョイント破断（§3）でしか起きない**。

05の推奨（方式A）は1しか実現できない。市販クローンも1止まりと推定される（§5）。**2と3を実装した瞬間に、このジャンルの「見世物」品質で頭ひとつ抜ける**——これが本調査の設計上の含意である（確度: 分析）。

## 3. ジョイント破断の定番実装（Box2D v3）

### 3.1 公式API（一次確認済み）

Box2D v3 は破断用の力取得APIを一次サポートする（[Joint API](https://box2d.org/documentation/group__joint.html)、確度: 高）:

```c
b2Vec2 b2Joint_GetConstraintForce( b2JointId jointId );  // 拘束力 [N]
float  b2Joint_GetConstraintTorque( b2JointId jointId ); // 拘束トルク [N・m]
void   b2DestroyJoint( b2JointId jointId );
```

公式マニュアルは「You can use reaction forces to break joints or trigger other game events. See the sample `BreakableJoint` for more details.」と明記（[Simulation doc](https://box2d.org/documentation/md_simulation.html)、確度: 高）。

### 3.2 BreakableJoint サンプルの破断ロジック（雛形）

[sample_joints.cpp](https://github.com/erincatto/box2d/blob/main/samples/sample_joints.cpp) の該当部（確度: 高、ソース確認）:

```cpp
// every frame:
b2Vec2 force = b2Joint_GetConstraintForce( m_jointIds[i] );
if ( b2LengthSquared( force ) > m_breakForce * m_breakForce )
{
    b2DestroyJoint( m_jointIds[i], true );
    m_jointIds[i] = b2_nullJointId;
}
```

- 破断閾値 `breakForce` はサンプル内でスライダ調整（0〜10000N）。**閾値は実プレイ調整項目**であり定数化しない設計が正しい。
- v2時代の定番（iforce2d流 `GetReactionForce(inv_dt)`）と同じ思想だが、v3では専用APIになった。ジョイント/ボディの破壊は**ステップ中ではなくステップ完了後**に行う（v3はイベントをポーリング型で返すため、`WorldStep()` 直後の破断チェックが自然に安全になる）（確度: 高。[iforce2d joints overview](https://www.iforce2d.net/b2dtut/joints-overview) / Box2D docs）。

### 3.3 実戦パターン（スパイク・ノイズ対策 + 「軋み」への転用）

ソルバーは単発フレームで力のスパイクを出すことがあるため、素の閾値比較だと「触れた瞬間ポキン」が起きる。定番の強化は:

1. **EMA平滑化**: `stress = 0.85*stress + 0.15*raw` のような指数移動平均で1〜2フレームのスパイクを無視。
2. **連続超過カウント**: 閾値超過が3〜5ステップ続いたら破断（ヒステリシス）。
3. **合成応力**: revolute鎖では「曲げ」がトルクに、「引張/せん断」が力に出るため `raw = |F|/breakForce + |T|/breakTorque` の合成が扱いやすい。
4. **軋み演出への直結**: stress 0.6〜1.0 の帯域を、そのまま (a) 軋みSFXの音量/ピッチ、(b) 線分の色（白→黄→赤）、(c) 微振動・粉パーティクル、(d) ハプティクス弱連打にマッピングする。**Poly Bridge の stress 可視化（応力を色で表示し、限界で破断）と同じ構造**で、03のjuice要件「軋み」が物理値から無料で手に入る（確度: 高（構造）/ 中（具体パラメータ）。[Poly Bridge stress view mod](https://github.com/56kyle/StressTypeVisualization) / [Poly Bridge manual](http://cdn.akamai.steamstatic.com/steam/apps/367450/manuals/ManualPFDable.pdf)）。

これらは推測ではなく、破壊可能構造物ゲームの一般慣行（Poly Bridge系）＋Box2D公式サンプルの合成である。

### 3.4 Phaser Box2D での可用性（ポート固有の確認結果）

リポジトリを直接検索して確認（確度: 高）:

- `b2Joint_GetConstraintForce` / `b2Joint_GetConstraintTorque` / `b2DestroyJoint` はすべて `src/main.js` からエクスポート済み。
- revolute の `enableSpring / hertz / dampingRatio / enableLimit / SetLimits`、weld の `linearHertz / angularHertz`、distance/prismatic/wheel のばね一式も移植済み。
- 注意: ポートの `b2DestroyJoint(jointId)` は引数1個（C最新版の `wakeBodies` 第2引数は無い）。破断後に `b2Joint_WakeBodies` 相当の起床が必要ならボディを明示的に起こす（確度: 中〜高）。

## 4. 性能特性: 描線Nセグメント化のコスト

### 4.1 シーン予算の試算

| 項目 | 数量 |
|---|---|
| 描線1本（セグメント上限32と仮定） | dynamic body ≤32、capsule fixture ≤32、revolute/weld joint ≤31 |
| 車 | body 3（シャシー+車輪2）、wheel joint 2 |
| 地形 | static chain 1（bodyほぼノーコスト） |
| **合計（最悪ケース）** | **動的 body ~35 / joint ~33 / fixture ~40** |

複数ストローク許可でも body 100 / joint 100 を超えない設計にできる。

### 4.2 ベンチマークとの突き合わせ

- **Box2D（C版）公式サンプル実績**: Bridge=板160枚+revolute 160本、Driving=橋20枚+車+地形一式をインタラクティブ実行。scissor lift サンプルは sub-step 8（480Hz相当）で安定動作（確度: 高、ソース/[docs](https://box2d.org/documentation/md_simulation.html)）。
- **JS/WASM系の規模感**: Matter.js（純JS・最遅級）でもデスクトップで4,500 bodies/38fps、Rapier(WASM)は7,500 bodies/60fps（[dev.to ベンチ記事](https://dev.to/jerzakm/this-little-known-javascript-physics-library-blew-my-mind-57oo)、確度: 中——ハードウェア非開示）。[napejs.org の対話ベンチ](https://napejs.org/benchmark.html)も同傾向。
- Phaser Box2D 自体の公表数値は無い（[phaser.io/box2d](https://phaser.io/box2d) / [告知記事](https://phaser.io/news/2024/12/announcing-phaser-box2d) に具体値なし、確度: 高）。純JS移植なので Planck.js（Box2D v2系純JS）と同オーダーとみるのが妥当（確度: 中、推測）。

**推定**: 中級Android WebViewがデスクトップ比で5〜10倍遅いと仮定しても、有効負荷が「Matter.jsが38fpsを出せる規模の 1/100 以下」なので、**物理ステップは1〜2ms未満に収まる公算が大きい**。60fps予算16.6msの支配項は物理ではなく描画・GCになる（確度: 中、複数ベンチからの外挿。実測はスパイクで確定させる）。

### 4.3 数（N）よりも効く安定性要因

1. **質量比**: 車が重く（例: シャシー質量 M）、セグメントが軽い（m）と M/m が大きいほどジョイントが伸び・発振する。Box2D v3 のメンテナは「iterative な SoftStep solver は sub-stepping と soft constraint で高質量比・長い鎖を扱える」と明言（[Erin Catto @X](https://x.com/erin_catto/status/1908939691321344426)、確度: 高）。実務指針: 線の総質量を車の0.3〜1.0倍程度に確保し、セグメント単体との質量比を ~30:1 以内に抑える（確度: 中、慣行）。
2. **sub-step**: 「長いジョイント鎖は sub-step を増やすほど伸びが減る」（公式docs）。既定4 → たわみ橋では6〜8を試す（確度: 高）。
3. **セグメント数は少ないほど安定**: N=10〜16 で始め、見た目の滑らかさは描画スプラインで補間する（物理セグメントと描画頂点を分離）。N=40 は性能ではなく安定性・チューニング難度の理由で避ける（確度: 中、分析）。
4. **自己衝突オフ**: 同一ストローク内は `filter.groupIndex = -strokeId` で衝突無効、`collideConnected=false`。カプセル形状なら継ぎ目のゴースト衝突も出にくい（確度: 高、Box2D docsの一般則）。

### 4.4 60fps維持の目安（結論）

- **ジョイント数 30〜100本は、2026年の中級Android（Snapdragon 6xx/Helio G系）WebViewで問題になる負荷ではない**（確度: 中。直接の公開WebView実測は見つからず——だからこそ着手週スパイクの必須項目）。
- 05が示した緩和策（resolution scale等）は描画側の話であり、本方式採用による物理側の追加リスクは小さい。

### 5.2 解剖の結論

- **市販クローン群の描線は「単一剛体（EdgeCollider/Polygon + 1 Rigidbody）」が支配的実装**と判断する。根拠: (1) Unityエコシステムの定番レシピと市販テンプレートが全てこの構成、(2) どの商品説明も曲げ・破断を売りにしていない（sturdy=頑丈と明記する例すらある）、(3) 壊れる主体は車（ragdoll）側に置かれている。**確度: 中**（ソースコード非公開のため状況証拠による推定。動画フレーム解析による直接確認は未実施）。

## 6. 推奨アーキテクチャと実装パターン

### 6.1 推奨: 3層構造の描線物理

```
[入力層]   pointer点列 → RDP間引き → 等間隔リサンプル（セグメント長 0.5〜0.8m相当）
[物理層]   カプセルセグメント×N（N=8〜24, 上限32）
           + revolute joint（enableSpring: hertz 4〜8, dampingRatio 0.6〜0.8,
             enableLimit: ±0.2〜0.4rad, collideConnected=false）
           + 破断: |F|/Fbreak + |τ|/Tbreak のEMA > 1.0 で b2DestroyJoint
[描画層]   物理セグメントの位置からCatmull-Rom/スプラインで滑らかな1本線を再構成して描画
           （破断点で描画パスを分割。折れ口にギザギザ表現）
```

- 剛性の初期値は Box2D 公式 Bridge サンプル（hertz 2.0 / damping 0.7）を「柔らかすぎる橋」、Driving の車サス（hertz 5.0 / damping 0.7）を参考帯として、**hertz 4〜8 から調整開始**（確度: 中——公式サンプル値からの外挿。最終値は実機チューニング）。
- weld(angularHertz) 版は「revolute+spring で角度が暴れる／もっと板らしくしたい」場合の差し替え候補。両者はジョイント生成関数の差し替えだけで A/B できるよう抽象化しておく。
- **フォールバック層**: 同じ入力層・描画層のまま、物理層だけを「単一compound（05方式）」に差し替えられる構成にする。スパイク失敗時・低性能端末向け設定として保険になる。

### 6.2 Phaser Box2D 実装パターン（コード例）

```ts
// 1) stroke -> segmented capsule chain
const pts = resample(rdpSimplify(rawPoints, EPS), SEG_LEN);
const bodies: b2BodyId[] = [];

for (let i = 0; i < pts.length - 1; i++) {
    const mid = midpoint(pts[i], pts[i + 1]);
    const bodyDef = b2DefaultBodyDef();
    bodyDef.type = b2BodyType.b2_dynamicBody;
    bodyDef.position = pxmVec2(mid.x, mid.y);
    const bodyId = b2CreateBody(worldId, bodyDef);

    const shapeDef = b2DefaultShapeDef();
    shapeDef.density = LINE_DENSITY;        // 車との質量比 <= ~30:1 を目安に
    shapeDef.friction = 0.8;
    shapeDef.filter.groupIndex = -strokeId; // same-stroke self-collision off

    const half = toLocalHalfVec(pts[i], pts[i + 1]);
    const capsule = new b2Capsule(neg(half), half, LINE_RADIUS);
    b2CreateCapsuleShape(bodyId, shapeDef, capsule);
    bodies.push(bodyId);
}

// 2) bendy joints
type StrokeJoint = { id: b2JointId | null; stress: number };
const joints: StrokeJoint[] = [];

for (let i = 0; i < bodies.length - 1; i++) {
    const jd = b2DefaultRevoluteJointDef();
    jd.bodyIdA = bodies[i];
    jd.bodyIdB = bodies[i + 1];
    jd.localAnchorA = endAnchorA(i);
    jd.localAnchorB = endAnchorB(i + 1);
    jd.enableSpring = true;        // bending stiffness (the "sag")
    jd.hertz = 6.0;
    jd.dampingRatio = 0.7;
    jd.enableLimit = true;         // max fold angle before it reads as "broken"
    jd.lowerAngle = -0.3;
    jd.upperAngle = 0.3;
    jd.collideConnected = false;
    joints.push({ id: b2CreateRevoluteJoint(worldId, jd), stress: 0 });
}

// 3) after WorldStep(): creak + break
function updateStress(dtStepped: boolean) {
    if (!dtStepped) return;
    for (const j of joints) {
        if (!j.id) continue;
        const f = b2Joint_GetConstraintForce(j.id);
        const t = b2Joint_GetConstraintTorque(j.id);
        const raw = Math.hypot(f.x, f.y) / BREAK_FORCE + Math.abs(t) / BREAK_TORQUE;
        j.stress = 0.85 * j.stress + 0.15 * raw;      // EMA: ignore solver spikes

        applyCreakJuice(j, j.stress);                 // SFX pitch/vol, color, haptics @0.6..1.0

        if (j.stress > 1.0) {
            b2DestroyJoint(j.id);                     // partial collapse: chain splits here
            j.id = null;
            spawnSnapJuice(j);                        // crack SFX, debris, camera trauma
        }
    }
}
```

（フィールド名 `lowerAngle/upperAngle`、アンカー系ヘルパはポートの実シグネチャに合わせて要最終確認。`b2Joint_GetConstraintForce` 等のエクスポートは確認済み。）

### 6.3 固定タイムステップと再現安定性への影響

- Phaser Box2D の `WorldStep()` は**アキュムレータ式固定タイムステップを内蔵**（既定 fixedTimeStep=1/60、subStepCount=4、借用時間2フレーム分・キャッチアップ上限2回）——ソース直接確認（確度: 高）。Gaffer on Games 流のレンダリング補間は含まれないため、必要なら自前補間を足す（[Fix Your Timestep!](https://gafferongames.com/post/fix_your_timestep/)）。
- **決定論**: C版 Box2D は v3.1 以降「algorithmic / multithreaded / cross-platform」の3レベルの決定論を既定で満たす（fast-math回避・自前atan2等による。[Determinism post](https://box2d.org/posts/2024/08/determinism/) / [FAQ](https://box2d.org/documentation/md_faq.html)、確度: 高）。
- ただし **JSポートにこの保証がそのまま乗るかは未検証**。JSの四則演算はIEEE754で決定的だが、`Math.sin/cos/atan2` はブラウザエンジン間で結果が異なり得る。よって**「同一端末・同一ブラウザ内の再現」は期待でき、「端末間・ブラウザ間の再現」は保証されない**前提で設計する（確度: 中、分析）。00調査の決定論懸念への回答: リプレイ/ゴースト機能は「入力再生」ではなく「位置サンプル再生」で実装すれば、この差異の影響を受けない。
- ジョイント破断は決定論をさらに敏感にする（破断1本の有無で以後が分岐）。同一端末内でも、破断判定は必ず物理ステップに同期（可変レンダフレームではなく `WorldStep` 消費ごと）で行うこと。

### 6.5 着手週の実機スパイク設計（このレポートの実務出力）

**目的**: 「方式選定を主観の手触り+実測fpsで1日以内に確定」。

- **シーン**: 崖ギャップ8m、描線1本（固定シード: 同一点列をコードで注入）、車=シャシー+wheel joint×2（Drivingサンプル値）、60fps固定+subStep4。
- **バリアント**（実行時トグルで切替）:
  - A: 単一compound（05方式） — 対照群
  - B: revolute鎖（ばね無し、limit無し）
  - C: revolute鎖 + spring(hertz 4/6/8) + limit ±0.3rad
  - D: weld鎖（angularHertz 3/6）
  - 各 N=8 / 16 / 24 / 32
- **計測**: (1) `WorldStep` 実測ms（p50/p95）、(2) 描画込みfps、(3) ジョイント最大伸び（アンカー間距離）、(4) 車が渡り切れるかの成功率10試行、(5) 破断ON時の「見世物」主観評価（録画して並べる）。
- **端末**: Snapdragon 6xx / Helio G 系の実機WebView（Capacitorシェル）+ Mac Chrome（基準値）。
- **合格基準**: 実機で p95 step ≤ 4ms かつ 60fps維持、C or D のいずれかで「荷重たわみが視認でき、破断が誇張なく決まる」こと。不合格時のみ A（compound）+ 演出たわみ（描画層のみ変形）へ後退。
