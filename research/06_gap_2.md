# レベルデザイン文法とコンテンツ量産・クリア可能性自動検証パイプライン — ギャップ補完調査

> 調査日: 2026-07-07 / 調査者: ギャップ補完調査員（Claude）
> 対象: 線描き物理パズル（Draw Bridge / Happy Glass / Love Balls / Brain Dots / Draw Climber 系）のレベルデザイン文法、量産手法、クリア可能性の自動検証
> 前提: 本作は「線を引いて橋を作り車を走らせる」タイプB×タイプAハイブリッド。技術スタックは Phaser 4 + Phaser Box2D（Box2D v3 JSポート）で内定済み（`research/05_tech_stack.md`）
> 確度表記: 高（複数ソースで裏取り済み）/ 中（単一ソースまたは強い状況証拠）/ 低（推測・分析）

## エグゼクティブサマリー

1. **ギミック語彙はジャンル全体で約12類型に収束する**。Happy Glass の公開分析だけでも「可動物・回転体・熱面（水を消す）・描画禁止区域・複数ゴール・複数蛇口」等が確認でき、Draw Bridge 系は「対向車・崖・砲弾・落下貨物・複数車両・一筆書き制限」を加える。ジャンル定番の「箱＝25面＝新メカニクス1つ」（Cut the Rope）文法と合わせ、本作のギミックカタログは調査ベースで構築可能（確度: 高）。
2. **ただし「何面で何が解禁されるか」の正確な対応表は公開情報に存在しない**。攻略サイトは解法画像のみでメカニクス解禁面を記録しておらず、体系的な解禁ロードマップは競合実機プレイで自前採取する必要がある（確度: 高、= 本調査の残ギャップ）。
3. **「直線1本」支配戦略の無効化は6系統の制約で実現できる**: (a) インク予算＋スター経済、(b) 描画禁止区域、(c) インク予算を超える谷幅（中間支点の強制）、(d) 動的荷重（壊れやすい車・落下物・砲弾）、(e) 描線を破壊するハザード（レーザー等）、(f) 複数車両の衝突回避。学術側でも「貪欲戦略を罰するレベル生成」（Deceptive Level Generation, 6欺瞞カテゴリ）と「構造的弱点の保護」（Stephenson & Renz）が同型の問題として定式化済み（確度: 高）。
4. **市場基準は「ソフトローンチ時500面＋10〜15面ごとに新メカニクス＋運用で週20面追加」**。PocketGamer.biz（2025-26）が「D30 5〜7.5%を出すには soft launch 時点で 500面＋LiveOps＋メタ進行の萌芽が必要」と明言。Mobile Game Doctor は「週20面リリース（≒プレイ5時間分）」「良いツールはデザイナー生産性を5倍にする」と提示（確度: 高）。
5. **物理パズルの機械生成は「生成→ボット検証→難易度推定」の三点セットが定石**。Angry Birds 系研究は行単位の構造生成＋支持検証＋確率テーブルで多様な構造を作り、エージェント実プレイで解の存在を確認する。Match-3 では cVAE 生成＋ヒューリスティックボット30回試行の自動検証ループで有効レベル率 51.4% を達成。難易度推定は「プレイヤーデータ＋ボットデータ併用」が最良で、新規レベル（コールドスタート）ではボットデータが必須（確度: 高）。
6. **Box2D v3.1（C版）は クロスプラットフォーム決定論を達成済みだが、Phaser Box2D（JSポート）はその保証を持たない**。ポートのコアが JS 組み込みの `Math.sin/cos/atan2` を使用していることをソースコードで確認（`src/include/math_functions_h.js`）。ECMAScript は超越関数のビット一致を規定しないため、**同一エンジン（同一Node/V8）内では再現可能、ブラウザ間ではビット一致は保証されない**（確度: 高、コード確認済み）。
7. **本作はジャンル特性上、決定論問題が極小化できる**。「描いてから走らせる」ゲームは実行中の入力がゼロで、リプレイ＝「レベルデータ＋描線ポリライン＋tick数」だけで完結する。CI を固定 Node バージョン（=単一V8）に固定すれば**ビット一致のゴースト解検証が可能**、実機側の差異は「±kティック・±ε座標」の許容帯（tolerance band）で吸収するのが現実解（確度: 高）。
8. **FTUE定石は「初回30〜60秒で継続判断される」「1面目は失敗がほぼ不可能」「テキストチュートリアル禁止」「45秒以内にコア快感」**。新メカニクスは「失敗させない教習面→試験面→ブレザー（息抜き）面」のループで導入する（確度: 高）。

## (1) ギミック語彙の網羅カタログ

### 1.1 作品別に確認できたギミック

#### Happy Glass（Lion Studios, 2018 / 500面）

公開されているゲームデザイン分析（Olin Olmstead, Medium）と攻略サイト群から確認できた語彙（確度: 高）:

| ギミック | 説明 | 支配戦略への効き方 |
|---|---|---|
| インク予算＋スター | 描いた量が少ないほど星が多い（1〜3星）。星3は広告視聴でも補完可 | 「雑な大きい線」を経済的に罰する |
| 不動オブジェクト | 静的地形・壁 | 水路の強制 |
| 可動オブジェクト | 押せる・倒せる物体（線の重さで倒すギミックあり） | 「描いた線の質量」自体を道具化 |
| 回転体（spinny thing）/ ファン | 回転する羽根・ファンが水や線を弾く | タイミング・配置制約 |
| 熱面（hot surface） | 触れた水を蒸発させる（キャンドル等） | 経路の迂回を強制 |
| 描画禁止区域（non-drawable area） | 線が引けない領域 | 幾何的に直線を禁止 |
| 複数グラス | ゴールが複数 | 1本の線では両立不可能に |
| 複数蛇口（multiple spouts） | 水源が複数 | 分岐処理を強制 |
| グラス向き調整 | ゴール自体が傾く/動く | 最終アプローチ角の制約 |
| （攻略サイト言及）動く足場・テレポータルなど | 300面分の解法集で言及 | 時間依存の解を強制 |

#### Love Balls（SuperTapx, 2018）

| ギミック | 説明 |
|---|---|
| フェンス・仕切り | 2つのボールを分断する静的障害 |
| 落下死（非グリーン床） | 緑=安全地帯、それ以外の床からは落下 → 受け皿（ネット）描画を強制 |

#### Draw Bridge 系（Eureka Studio「Draw Bridge!」2021 / Bravestars「Draw Bridge Puzzle」/ GameLord 3D「Draw Bridge Games: Car Bridge」）

本作の直接競合。確認できた語彙（確度: 中〜高。複数タイトルの寄せ集めである点に注意）:

| ギミック | 説明 |
|---|---|
| 崖・谷（cliff / gap） | 基本地形。谷幅がレベルの主パラメータ |
| 対向車・暴走車（reckless cars） | 動くハザード。タイミング回避を強制 |
| 砲弾・爆発物（explosive bullets） | 橋や車を破壊する飛来物 |
| 落下貨物（falling cargo） | 上から落ちてくる動的荷重。橋の強度を試す |

#### Brain Dots（Translimit, 2015）

- ギミック語彙は意図的に**極小**（静的地形の形状バリエーションのみが主）。「青玉と赤玉をぶつける」1目標に対し、地形の凹凸・分断・高低差だけで500面以上を編んだ。代わりにペン収集メタと UGC（累計300万ステージ）でコンテンツを外部化（確度: 中）。

#### Draw Climber（Voodoo, 2020）— 地形語彙側の参照

攻略情報から確認できる**地形語彙**（= 描く形を変えさせる装置）: 平地（直線・コンパクト形状が最速）/ 階段・段差（L字・V字・Z字が有利）/ 壁・高台（縦長矩形）/ 深い溝（リーチの長い形状）。「地形を読んで形を描き替える」ことがゲーム性の核で、**地形バリエーションが乏しいと直線が支配戦略化する**ことが攻略メタで実証されている（確度: 高）。

### 1.2 登場順（解禁面）について — 重要な調査限界

- **攻略サイト・Wiki は解法画像を並べるだけで、「レベルNでギミックXが初登場」という対応表を公開していない**。Happy Glass の Medium 分析も「新しい障害物とメカニクスが導入され続ける」と述べるにとどまり、面番号との対応は記録していない（確度: 高 = 「無い」ことの確認）。
- ジャンル文法として引用可能な確定情報は **Cut the Rope の「1箱=25面、箱ごとに新メカニクス1つ、最初の Cardboard Box は教習用」**（確度: 高）と、**市場側の「新メカニクスは10〜15面ごと」**（PocketGamer.biz、確度: 高）。
- → **解禁ロードマップの実数値は、競合3本（Draw Bridge 系2本＋Happy Glass）の最初の100面を実機プレイして採取するのが唯一の方法**。1本あたり2〜3時間で採取可能（各面15〜40秒 × 100面 + 記録）。

出典: [Cut the Rope Wiki: Cardboard Box](https://cuttherope.fandom.com/wiki/Cardboard_Box) / [PocketGamer.biz: What happened to hypercasual](https://www.pocketgamer.biz/what-happened-to-hypercasual-the-markets-evolution-over-the-past-year/)

## (2) 支配戦略（直線1本）を無効化するレベル制約パターン

### 2.1 競合実例からの帰納

| パターン | 実例と機序 | 確度 |
|---|---|---|
| **インク上限＋星経済** | Happy Glass / Love Balls。直線でもクリアは可能だが星3を逃す。「クリア≠最適」の二層構造で、直線は「損な解」になる。コイン→ペン/車アンロックに接続すると経済圧が持続する | 高 |
| **谷幅 > 安定スパン** | Draw Bridge 系。「弱い線は崩れる」物理により、支持なしの長スパン直線は自重＋車重でたわむ/折れる。中間支点（G1地形の突起）に載せる形状を強制 | 高（機序は物理的必然） |
| **描画禁止区域** | Happy Glass。最短経路上に禁止区域を置けば直線は物理的に引けない | 高 |
| **動的荷重** | Draw Bridge の落下貨物・砲弾、壊れやすい車。静的には成立する直線が、衝撃荷重で崩壊/車が破損。「クリア判定=走行完了」なので描いた瞬間の成立では不十分 | 高 |
| **描線破壊ハザード** | Love Balls のレーザー。特定領域を通る線はその場で無効化 | 高 |
| **複数車両の衝突回避** | Draw Bridge。1本の直線に2台載せると追突する。分離帯・待避形状を強制 | 中 |
| **スター/収集物の配置** | Cut the Rope 文法。星を放物線上・谷底に置き、「クリアだけなら直線、星3なら曲線」を作る | 高（ジャンル文法として） |
| **地形ローテーション** | Draw Climber の教訓の裏返し。平地→階段→水面（Scribble Rider）のように「前の形が通用しない地形」を10〜15面周期で回す | 中（分析） |

### 2.2 学術研究からの裏付け — 「貪欲戦略を罰する」は定式化済み

1. **構造的弱点の保護（Stephenson & Renz, AAAI 2016）**: Angry Birds 用レベル生成器は、「取り除くと大量のオブジェクトが崩れるブロック（weak point）」を特定し、(a) 手前に保護スタックを積む、(b) 支持ブロックを追加する、(c) 素材を石に変える、の3手段で**貪欲な一撃（greedy shot）の有効性を意図的に下げる**。これは「直線1本」対策のアルゴリズム版そのものであり、本作では「最短スパンの直線が通る空間に禁止区域/ハザード/たわみ荷重を自動配置する」ジェネレータ後処理として移植できる（確度: 高、論文本文を直接確認）。
   出典: [Procedural Generation of Levels for Angry Birds Style Physics Games (PDF)](https://matthewstephenson.info/papers/Procedural%20Generation%20of%20Levels%20for%20Angry%20Birds%20Style%20Physics%20Games.pdf)
2. **欺瞞レベル生成（Deceptive Level Generation for Angry Birds, IEEE CoG 2021)**: 「次の一手の最善」だけを探すエージェント（=貪欲プレイヤー）が失敗する6カテゴリの欺瞞（転がり/落下物の利用、経路の事前クリア、素材強度の読み、非貪欲行動の強制など）を生成時に付与し、**生成と同時に想定解も出力**する。「意図した解では解ける／意図しない自明解では解けない」の両方をメトリクス化しており、本作の「直線ボットは失敗しなければならない」ゲート（§4.4）の理論的裏付けになる（確度: 高）。
   出典: [Deceptive Level Generation for Angry Birds (arXiv:2106.01639)](https://arxiv.org/abs/2106.01639)

### 2.3 本作への含意（分析、確度: 中〜高）

- 支配戦略対策は**単一の銀の弾丸ではなく「経済（G2）×幾何（G1/G3）×動力学（G5/G6）の重ね掛け」**で行う。序盤はG2のみ（直線でもクリアできるが星が減る）、中盤からG1/G3（直線が物理的に不成立）、後半でG5/G6（動的に破綻）へ移行させると、学習曲線と対策強度が同期する。
- **検証可能性が高いのがこの設計の最大の利点**: 「直線1本ボット」は自明に実装できる（両崖の縁を結ぶ線分を1本描くだけ）。よって「このレベルは直線で解けてはならない」を CI の自動テストにできる（§4.4）。

### 3.2 レベル制作パイプラインの業界標準（Mobile Game Doctor、確度: 高）

- **難易度曲線は「ノコギリ波（saw curve）」**: 学習/息抜き面 → 積み上げ面 → クライマックス（ボス）面 → 新メカニクスの学習面…の繰り返し。単調増加は禁止。
- **制作6フェーズ**: ①コンセプト（紙/ドキュメント）→ ②社内テスト＋実プレイヤーのユーザビリティテスト → ②B 挙動ベース微調整 → ③ **AI/MLオートプレイでのバランシング** → ③B 小規模コホートでのA/Bテスト → ④ 運用中の継続リバランス。
- **難易度KPI**: 勝率/失敗率、敗北時の残り手数、目標達成率。改善は「レイアウト・障害物・手数・スポーン率」の調整で行う。
- **ツール投資**: 「動くツールはデザイナーを5倍速にする」「実機で即プレイできるエディタが最良」「レベルデザイナーは複数人置いてデプロイリスクを下げる」。

### 3.3 物理パズルのプロシージャル/パラメトリック生成 — 先行研究

### 3.4 本作の量産戦略への翻訳（分析、確度: 中）

物理橋ゲームのレベルは本質的に**少数パラメータの地形＋ギミック配置**であり、以下の3層で300〜500面を量産できる:

1. **テンプレート層（手作り、〜30種）**: 「谷1つ」「二段谷」「高低差登坂」「天井制限」「島渡り」等の地形トポロジーを人間が設計。各テンプレートはギミックスロット（G4〜G9 の設置点）を持つ。
2. **パラメータ変奏層（半自動、テンプレート×10〜20変奏）**: 谷幅/インク予算比、支点の位置と数、ハザードのタイミング窓、車重・車速、スター位置をパラメータ化して変奏を生成。Stephenson 式の妥当性検査（支持成立・到達可能性・重なり禁止）で足切り。
3. **検証・難易度層（全自動）**: §4 のボット検証パイプラインで (a) 可解性、(b) 支配戦略不成立、(c) 難易度スコア（ボット群の成功率・試行回数中央値）を算出し、ノコギリ波曲線に沿って面順に並べ替える。

エディタ投資の相場観: 業界の教訓は「**ゲーム本体のランタイムを流用した実機プレイ可能なエディタ**が最小で最良」（Mobile Game Doctor）。本作は Web ランタイムなので、**ゲーム本体＋デバッグUI（地形頂点編集・ギミック配置・JSONエクスポート・解の録画ボタン）で済み、専用ツール開発は1〜2週間規模**と見積もる（分析、確度: 中）。「ツールで5倍」「週20面」を合わせると、ツール整備後は1人で週15〜20面（検証込み）が現実的な目標値。

## (4) クリア可能性の自動検証 — 技術検証

### 4.1 Box2D v3 の決定論の実態（確度: 高）

Box2D 公式ブログ（Erin Catto, 2024-08）と FAQ から:

- **v3 はデフォルトで3段階の決定論を提供**: ①アルゴリズム的決定論（乱数不使用、同一マシンで run-to-run 一致）、②**マルチスレッド決定論**（2スレッドでも8スレッドでも同一結果。ワーカーのビット配列を main で OR 合成して順序を保存）、③**クロスプラットフォーム決定論（v3.1〜）**: x64/ARM、MSVC/GCC/Clang 間で一致。
- 達成手段: fast-math 不使用 / **FMA無効化（`-ffp-contract=off`）** / `sinf`・`cosf`・`atan2f` を**独自の近似実装に置換**（`sqrtf` はSSE命令が決定的なのでそのまま）。
- **CI に "Falling Hinges" 決定論テスト**があり、スリープまでのステップ数と全 transform のハッシュ値（例: `0x5e70e5fe`）をプラットフォーム間で比較している。→ **「終了状態ハッシュの一致」は本家も使う実証済みのCIパターン**。
- **ロールバック決定論は無い**（内部状態の巻き戻し機構なし）。リプレイは「初期状態から再シミュレーション」で行う。

出典: [Box2D: Determinism](https://box2d.org/posts/2024/08/determinism/) / [Box2D FAQ](https://box2d.org/documentation/md_faq.html)

### 4.2 Phaser Box2D（JSポート）の決定論 — コード検証結果（確度: 高）

公式は「Box2D は決定的な結果を生む。同一入力なら全クライアントが同一の物理状態を計算する」と謳う（[phaser.io/box2d](https://phaser.io/box2d)）。しかし **GitHub リポジトリのコード検索で以下を確認**した:

```
src/include/math_functions_h.js:  return new b2Rot(Math.cos(angle), Math.sin(angle));   // b2MakeRot
src/include/math_functions_h.js:  return Math.atan2(q.s, q.c);                           // b2Rot_GetAngle
src/revolute_joint_c.js:          const r = new b2Vec2(L * Math.cos(angle), L * Math.sin(angle));
```

- つまり **JSポートは本家 v3 の「独自トリガー関数」を移植しておらず、JS 組み込みの `Math.sin/cos/atan2` に依存**している（ポート元は v3.0 で、クロスプラットフォーム決定論が入った v3.1 以前）。
- ECMAScript は数値を IEEE 754 binary64 と規定し、**四則演算と `Math.sqrt` は決定的**だが、**超越関数（sin/cos/tan/atan2/exp/log）の結果はエンジン実装依存**。V8・SpiderMonkey は fdlibm 系実装に収束しつつあるが標準は要求しておらず、**Chrome と Firefox で二重振り子が数分で発散した実測報告**がある。
- **結論**:
  - **同一エンジン・同一バージョン内（例: CI の固定 Node.js = 固定 V8）**: run-to-run で**ビット一致が期待できる**（JSはシングルスレッド・fast-math なし・JITの最適化でも FP セマンティクスは保存される）→ **CI ゲートはビット一致で書ける**（確度: 高）。
  - **ブラウザ間・エンジン間（V8 vs JavaScriptCore vs SpiderMonkey）**: ビット一致は**保証されない**（確度: 高）。トリガー関数が使われるのは主に「角度からの回転生成（レベルロード時）」「revolute joint の角度制限計算」なので発散源は限定的だが、ゼロではない。
  - 対策オプション: (a) 実機側は許容帯検証にする（推奨、§4.4）、(b) ポートの `Math.sin/cos/atan2` 呼び出し（数カ所）を Box2D v3.1 の決定的実装の JS 移植に差し替える（MIT なのでフォーク可、工数小・確度: 中）、(c) ビット決定論が絶対要件になったら `@dimforge/rapier2d-deterministic`（WASM）へ移行 — **WASM は IEEE 754-2008 を厳密に規定するため、ブラウザ・OS・CPU が違ってもビット一致**が公式に保証されている（確度: 高）。

出典: [phaserjs/phaser-box2d（コード）](https://github.com/phaserjs/phaser-box2d) / [Math keeps changing (macwright)](https://macwright.com/2020/02/14/math-keeps-changing) / [Mozilla: fdlibm intent](https://groups.google.com/a/mozilla.org/g/dev-platform/c/0dxAO-JsoXI/m/eEhjM9VsAgAJ) / [Rapier: Determinism](https://rapier.rs/docs/user_guides/javascript/determinism/) / [@dimforge/rapier2d-deterministic](https://www.npmjs.com/package/@dimforge/rapier2d-deterministic) / [GameDev.net: Is JavaScript floating-point math deterministic?](https://www.gamedev.net/forums/topic/609592-is-javascript-floating-point-math-deterministic/)

### 4.3 固定タイムステップとリプレイの原則（確度: 高）

- 決定的リプレイの前提は**固定タイムステップ**（`b2World_Step(world, fixedDt, subStepCount=4)`）。レンダリングは accumulator パターンで補間し、物理は常に同じ dt・同じ回数で刻む（Gaffer on Games「Fix Your Timestep!」が定番。ロールバックや録画再生には固定ステップが必須というのが業界定説）。
- ゲームが決定的なら**リプレイ＝「初期状態＋全ティックの入力列」だけで任意の状態を再現できる**。
- 許容帯（tolerance band）: 自動運転シミュレーション検証の研究では「反復実行で安定した検証結果を得るには、アクター状態を**特定の許容誤差**まで一致させる。許容誤差は回帰を検出できるほど小さく、計算上の微小揺らぎで偽陽性を出さない程度に大きく」と定式化されている。ゲームQAでも Retro City Rampage（GDC）が「入力録画リプレイを自動テストに使う」実践例を公開。
- **本作のジャンル特性が効く**: 「描いてから走らせる」ため、**シミュレーション開始後のユーザー入力がゼロ**。リプレイデータは「レベルJSON＋描線ポリライン（正規化済み頂点列）＋最大tick数」で完結し、ティック単位の入力同期が不要。これは lockstep 系ゲームより桁違いに検証が容易（分析、確度: 高）。

出典: [Gaffer on Games: Fix Your Timestep!](https://gafferongames.com/post/fix_your_timestep/) / [Gaffer on Games: Floating Point Determinism](https://gafferongames.com/post/floating_point_determinism/) / [On Determinism of Game Engines used for Simulation-based AV Verification (arXiv:2104.06262)](https://arxiv.org/pdf/2104.06262) / [GDC: Automated Testing and Instant Replays in Retro City Rampage](https://www.gdcvault.com/play/1021825/Automated-Testing-and-Instant-Replays) / [Jakub Tomšů: Fixed timestep without interpolation](https://jakubtomsu.github.io/posts/fixed_timestep_without_interpolation/)

### 4.4 ゴースト解検証＋支配戦略ゲートの設計（本作向け提案、確度: 中〜高）

業界実例（The Division の Client Bots による「ミッション完走不能の自動検出」、Match-3 のボット30試行検証、Box2D 本家のハッシュ比較CI）を合成すると、本作の検証ゲート（`run-gate.sh` 相当）は次の5段で構成できる:

```
Gate 0: スキーマ検証        … レベルJSONが仕様に適合（zod等）。数秒/全面
Gate 1: 静的妥当性          … 到達可能性グラフ（崖端→ゴールが理論上接続可能）、
                              インク予算 ≥ 最小必要スパン×係数、ギミック参照の整合
Gate 2: ゴースト解リプレイ  … 各面に録画済み開発者解（1〜3本: 通常解・星3解・雑解）。
                              固定Node/固定dtで再生し「成功＋tick予算内＋星閾値達成」をassert。
                              同一プラットフォーム回帰は終了状態ハッシュ一致で検出（Box2D方式）
Gate 3: 支配戦略の否定      … 自明解ボット群（直線1本／最短2点結線／水平床いっぱい）を実行し、
                              「anti-dominant タグ付きの面では必ず失敗する」をassert
Gate 4: 難易度推定          … パラメタライズド・ボット群（描線にノイズ・支点ずらし）をN回実行し
                              成功率→難易度スコア化。曲線（ノコギリ波）からの逸脱を警告
```

- **Gate 2 の許容帯**: CI（固定V8）内はビット一致 or ハッシュ一致。エンジン更新・物理ライブラリ更新時は「成功/失敗の一致＋最終車両位置 ε=0.05m・成功tick ±30」の許容帯で判定し、パスしたら解を録画し直してハッシュを更新する2段運用（分析、確度: 中）。
- **解の録画は編集フローに組み込む**: エディタの「テストプレイ成功→この解を保存」ボタンで、レベル保存時に必ず1本以上の解が添付される制約（解のないレベルはスキーマ違反）。Deceptive Level Generation の「生成時に解も出力する」原則の人力版。
- **物理ライブラリのバージョンピン留め**: phaser-box2d / Node /（将来）Rapier のバージョンを lockfile で固定し、更新PRでは全面リプレイを必須ゲートにする。

### 4.5 難易度の実測とチューニング（確度: 高）

- ローンチ前: ボットデータのみが頼り（Kristensen+ 2024: コールドスタートではエージェントデータなしでは定数平均すら上回れない）。
- ローンチ後: 面ごとの failure rate / 試行回数 / 離脱率をイベント計測し、**「リテンションと収益の意味ある改善はレベル難易度進行の変更によってのみ可能」**（Mobile Game Doctor）という前提で週次リバランス。ノコギリ波からの逸脱面（連続失敗スパイク）を差し替える。

### 6.2 ギミック解禁ロードマップ案（10〜15面周期、Cut the Rope 箱文法、確度: 中=提案）

|---|---|---|
| Ch1 (1-15) | コア＋インク予算＋星（G2）、中間支点（G1） | 経済のみ（直線でもクリア可） |

### 6.3 自動検証ゲート（run-gate.sh 相当）設計方針

- **CI 環境**: 固定 Node.js（=固定V8）上でヘッドレス実行（Phaser のレンダラ不要。物理層は Phaser 非依存で独立ロード可能 — `05_tech_stack.md` §リスク3 と整合）。
- **ゲート構成**: §4.4 の Gate 0〜4。PRごとに Gate 0-3（全面で数分以内を目標）、nightly で Gate 4（ボット群 N 回試行の難易度推定）。
- **判定の二層化**: 同一環境回帰＝`stateHash` ビット一致 / 環境・ライブラリ更新＝許容帯（成功一致・最終位置 ε・tick ±k）→ 再録画。
- **支配戦略ゲート**: `anti-dominant` タグ面で「直線1本ボット」「最短結線ボット」が成功したら**ビルド失敗**。これが本作の『ゲームデザイン意図のテスト契約化』の中核であり、社内パイプラインの run-gate 思想をそのまま移植できる。
- **決定論リスクの残余**: phaser-box2d の `Math.sin/cos/atan2` 依存はブラウザ間ビット一致を破りうる。MVP では「CI決定論＋実機許容帯」で十分。リプレイ共有・非同期対戦等ビット決定論が必須の機能を将来入れる場合は、(a) ポートのトリガー関数差し替え、(b) rapier2d-deterministic 移行、の2択を再評価する。

### 6.4 エディタ投資方針

- ゲームランタイム同梱の**インゲームエディタ**（地形頂点編集・ギミックパレット・パラメータインスペクタ・即時テストプレイ・「成功→解を保存」・JSONエクスポート）に限定投資。専用デスクトップツールは作らない。
- 根拠: 「実機で即プレイできる包括的エディタが最良」「動くツールは5倍速」（Mobile Game Doctor）。Web ゲームはランタイム＝エディタプレビューが同一物なので、このジャンルで最も安価にこの理想を満たせる（分析、確度: 中〜高）。
- 目標生産性: ツール整備後、テンプレート法併用で **1人週15〜20面（検証ゲート通過込み）**。500面 ≒ 6〜8か月/1人 or 3〜4か月/2人（変奏自動化の効き次第）。

## 未解決の疑問（Open Questions）

1. **競合の正確なギミック解禁面**: 公開情報に存在しない。Draw Bridge 系2本＋Happy Glass の最初の100面の実機採取（各2〜3時間）が必要。
2. **phaser-box2d の run-to-run 決定論の実測**: 理論上は同一V8でビット一致のはずだが、未検証。スパイク（同一レベル×同一描線×1000回実行でハッシュ比較）を最初の技術検証タスクにすべき。
3. **許容帯の具体値（ε・±k tick）**: 本レポートの ε=0.05m / ±30tick は仮置き。実ゲームの感度分析（描線1px差での結果分布）で校正が必要。
4. **「直線1本ボット」の定義域**: 直線だけでなく「L字」「浅いV字」等の準自明解も否定すべきか、ゲームフィール実装後にプレイテストで判断。
5. **UGC（Brain Dots 型）を将来入れるか**: 入れるなら投稿時の自動可解性検証（本パイプラインの流用）が必須要件になり、レベルデータ仕様に投稿者解の添付を最初から入れておく価値がある。
