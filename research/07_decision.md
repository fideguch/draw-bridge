# 07. 統合意思決定ドキュメント — Draw Bridge 系プロジェクト最終決定

- 決定日: 2026-07-07
- 対象: 「線を引いて橋を作り、車を走らせる」物理ゲーム（iOS + Android、Web 併用）
- 入力: research/00〜06 全10レポート（末尾 §10 出典参照）
- 位置づけ: 本ドキュメントは全リサーチの結論を「決定」に変換したもの。以降の `/requirements_designer` → `designs/` 生成はこの決定を単一の正とする。

## 前提（プロジェクト憲法級）

1. **品質基準は商用レベル**。既存ローカル2作（Stadium Rush! / Glowgrid）はプロセスの参考であり品質基準ではない。「広告を付けても恥ずかしくない」ことを出荷条件とする。
2. **ゲーム性はゼロから発明しない**。Draw Bridge 系の成功パターン（累計5,000万DL超の Bravestars 版ほか）を模倣し、品質・脳汁・良心的な広告体験で磨き込む。
3. **iOS / Android 両対応**。環境非依存のコンポーネント設計（物理・ゲームロジックはエンジン/OS 非依存、広告・ハプティクス・計測はインターフェース抽象化）。
4. **最重要価値は爽快感**。「線を引く」「車が走り出す」「ゴール」の3場面に開発リソースを集中投下する。

## 2. 市場ポジショニング決定

### 2.1 決定: 「Draw Bridge サブジャンルの決定版」ポジション

**サブジャンル**: line-drawing 物理パズル（タイプB: 描いた物が構造物になる）に、走行フェーズ（タイプA: 描いた結果が動いて審判される）を持つハイブリッド。直接競合は Bravestars「Draw Bridge Puzzle」(53.5M DL)、GameLord 3D 版 (11.2M DL)、Eureka 版 (10M+ DL)。

**根拠**（01, 02）:
- 需要は実証済み: 2022年リリースの低予算クローンでも 1,000万DL・評価4.3前後を維持。隣接の Bridge Race（橋×レース）は 2.5億〜5億DL で「橋を架けて進む」ファンタジーの天井は高い。
- 供給は低品質: レビューの不満は「広告地獄（30秒毎・偽閉じボタン）」「直線1本で全クリできる平坦さ」「レベル使い回し」「進行消失バグ」に集中。**コンセプト自体への不満はほぼない**（5★比率72%）。
- サブジャンルの新規大型ヒットは2022年が最後 → **UAスケール前提にせず**、オーガニック（Web ポータル・SNS 動画映え）+ 高LTV設計で小さく黒字化する構造を前提にする。

### 2.2 差別化の優先順位（02 §g、06_gap_1 §5）

1. **物理の本物感と気持ちよさ** — 荷重で凹み・軋み・限界で折れる橋。市販クローンは全て単一剛体で、この体験を誰も提供していない（確度: 中の状況証拠だが、実装出来れば確実に体験差になる）。
2. **支配戦略の防止** — 「直線1本で全クリ」を経済×幾何×動力学の重ね掛けで排除（§3.3）。競合レビュー最大のゲーム性不満への直接回答。
3. **節度ある広告体験** — 競合の自滅点。リワード中心・インタースティシャル最小・広告除去IAPあり（§6）。
4. **共有適性** — 崩落の見世物・「あなたの橋」が毎回違う = TikTok/ショート動画のUAエンジン（Save the Doge の実証パターン）。
5. **Web 配信併用** — Poki / CrazyGames は draw 系の一等地。CPIゼロの需要検証チャネル + 追加収益（06_gap_4）。

### 2.3 ビジネスモデル: 最初からハイブリッドカジュアル設計

2026年に純広告ハイパーカジュアルは経済的に成立しない（CPI > 広告LTV）。目標KPI はハイブリッドカジュアル基準: **D1 35〜40%+ / D7 12〜20% / D30 5〜7.5% / ARPDAU $0.10〜0.50 / セッション6〜8分**（02 §g4）。

## 3. コアゲームデザイン決定

### 3.1 コアループ（1レベル = 10〜30秒）

```
レベル開始（車・地形・旗が静止表示。解くべき地形が一目で読める）
  → 画面をドラッグして線を1本描く（インク残量バーがリアルタイム減少）
  → 指を離す = 確定。線が物理化して落下・たわみ、同時に車が自動発進（0秒フィードバック）
  → 車が橋を渡る（橋は荷重で凹み・軋む。走行中の操作なし = 描いた線が審判される）
  → 旗に到達 = クリア演出（§4.3）→ 星評価（インク消費量基準）→ 次レベル
  → 失敗 = 崩落/落下/衝突の見世物 → 原因が視認できる形で 1秒以内にリトライ
```

- **操作は「描く」のみ**。走行中の運転操作・ブースト操作は入れない（成功パターン準拠、Voodoo「Intuitive」原則）。
- **一筆書き制約**: 1レベル1ストロークを基本とする（Bravestars/GameLord 準拠。パズル性の核）。複数ストローク面は後半チャプターの拡張ギミック（G10 描画順序）として導入余地を残す。
- 描いてから走らせる構造のため**実行中の入力がゼロ** — リプレイ・自動検証が「レベルJSON + 描線ポリライン + tick数」で完結する（06_gap_2 §4.3。品質保証戦略の土台）。

### 3.2 成功 / 失敗条件

| 判定 | 条件 |
|---|---|
| クリア | 車（車体基準点）がゴール旗の判定矩形に到達 |
| 失敗 | ①車が画面下限へ落下 ②車の転倒（屋根接地が一定時間継続）③壊れやすい車の破損（急落下・衝撃超過。中盤以降のギミック）④制限tick超過（タイムアウト、まれ） |
| 星評価 | インク消費量の閾値3段階（少なく描くほど高評価 = Happy Glass 方式）。クリア自体は星1でも可 |

- 失敗は**罰しない**（Not punitive）: ペナルティ・スタミナ・ライフ制なし。失敗演出は崩落の物理見世物そのものであり、暗転+短い残念音のみでリトライボタン即出現。
- 失敗の因果可読性: 崩落時に**折れた接合部をハイライト**し、どこが限界を超えたかが一目で分かる（軋み色変化 白→黄→赤 の延長。00 の「原因バナー」思想の物理版）。

### 3.3 レベル構造（障害物・コースバリエーション）

**ギミック語彙は12類型（G1〜G12）に確定**（06_gap_2 §1.3）:

| # | 類型 | 例 |
|---|---|---|
| G1 | 静的地形（谷幅・高低差・傾斜・中間支点） | 基本。谷幅がレベルの主パラメータ |
| G2 | インク予算 + 星経済 | 雑な線を経済的に罰する |
| G3 | 描画禁止区域 | 直線を幾何的に禁止 |
| G4 | 可動・回転オブジェクト（動く足場等） | 時間制約 |
| G5 | 破壊ハザード（レーザー・火・砲弾） | 経路否定・耐久 |
| G6 | 動的荷重（落下貨物・壊れやすい車） | 構造強度の検査 |
| G7 | 複数車両・対向車 | 同時性 |
| G8 | 複数ゴール/ソース | 分岐 |
| G9 | スイッチ・ポータル類 | 順序 |
| G10 | 描画順序依存（複数ストローク） | 順序 |
| G11 | 外力（風・コンベア） | 外力 |
| G12 | 収集スター/コイン配置 | 経路誘導 |

**解禁文法**: Cut the Rope 箱文法 + 市場標準に従い、**1チャプター=15面、チャプターごとに新ギミック1系統**（10〜15面周期）。チャプター内は「教習面（失敗させない）→積み上げ→クライマックス→息抜き」の**ノコギリ波難易度**。解禁順は 06_gap_2 §6.2 のロードマップ案（Ch1: コア+インク+支点 → Ch2: 広い谷+壊れやすい車 → Ch3: 禁止区域+スター → …）を初期値とし、競合実機採取（Open Question）で補正する。

**支配戦略（直線1本）の無効化は重ね掛けで**:
- 序盤: G2 のみ（直線でもクリア可、ただし星が減る）
- 中盤: G1/G3（谷幅 > 安定スパン、禁止区域で直線が物理的/幾何的に不成立）
- 後半: G5/G6/G7（動的に破綻）
- **「anti-dominant タグ面で直線1本ボットが必ず失敗する」を CI の自動テスト（Gate 3）にする**（06_gap_2 §4.4。ゲームデザイン意図のテスト契約化）。

**量産戦略**: テンプレート層（手作り地形トポロジー 〜30種）× パラメータ変奏層 × ボット検証層の3層で、ソフトローンチ級には 300〜500 面へ拡張可能な設計にする。レベルはすべて JSON データ（コード/データ分離）、**解のないレベルは保存できない**（保存時にゴースト解≥1本添付をスキーマで強制）。レベルエディタはゲームランタイム同梱のインゲームエディタに限定投資（専用ツールは作らない）。

### 3.4 序盤設計（FTUE、06_gap_2 §5）

- **最初の3面は合計60〜90秒で3連続成功**させる。L1 は狭い谷 + 潤沢インクで「どんな線でも渡れる」（10秒で初成功）。
- テキストチュートリアル禁止。学びはプレイに埋め込む（L2でインクメーター/星、L4で中間支点、L5で「星3には曲線が要る」初面、L8で壊れやすさ初体験）。
- 45秒以内にコア快感（描く→走る→ゴール演出のフルコース）を必ず1回体験させる。

### 3.5 1セッションの設計

| 項目 | 設計値 | 根拠 |
|---|---|---|
| 1レベル所要 | 10〜30秒（ゴール演出込みループ1周 ≤ 40秒） | 03 §4.6 |
| 失敗→リトライ | **1秒以内**・ワンタップ | Voodoo 原則 |
| クリア→Next活性化 | 1.5〜2.5秒（演出はスキップ可） | 03 §4.5-4.6 |
| ボーナス面 | 5面ごと（報酬5〜10倍。将来の RV 接続点） | 04 §1.2 |
| 想定セッション | 6〜8分 = 12〜20面 + アップグレード1〜2回 | 02 §g4 |
| セッション終端 | チャプター区切り + 「次の報酬まであとN」表示で次回起動の理由を作る | 03/04 |

## 4. 「脳汁」設計仕様 — 3場面の実装チェックリスト

> 数値は一次リサーチ（03、06_gap_1）の具体パラメータ。「初手で8割正しい出発点」であり、最終値は実機チューニングで確定する。すべての値は `TuningConstants`（+デバッグスライダUI）に集約し、マジックナンバー散在を禁止する（00 の規約踏襲）。

### 4.1 場面1: 線を引く瞬間（操作の快感の心臓部）

**必須（これが無いと成立しない）**
- [ ] **入力遅延ゼロ描画**: タッチ座標を同フレーム反映。スムージング（Catmull-Rom / 移動平均）は過去点のみに適用し、**線の先端は生タッチ座標に固定**（Swink 即時応答原則）
- [ ] 頂点間引き: 最小点間距離 4〜8px で頂点追加（RDP 間引きは物理化時）
- [ ] 線の太さ: 画面幅の 2〜3%（375pt幅で 8〜12pt）、丸キャップ・丸ジョイント、高コントラスト単色 + 外周1〜2px の濃色ボーダー
- [ ] **描画中ループ音**: ペン/マーカー音。描画速度→音量 0.3→1.0 / ピッチ 1.0→1.2 連動。**指が止まったら音も止まる**。開始/停止 30〜50ms フェード。ストローク毎に基準ピッチ ±5% ランダム化
- [ ] **インク残量バー**: リアルタイム減少。>50% 緑 / 20〜50% 黄 / <20% 赤+点滅(300ms)。枯渇時「カスッ」空振り音 + バー横シェイク(4〜6px, 150ms) + warning ハプティクス
- [ ] **線確定ポップ**: 指を離した瞬間 scale 1.0→1.06→1.0（120ms, ease-out-back）+ 確定音（50〜120ms「コトッ」）+ ハプティクス iOS `.light`（`prepare()` 済）/ Android `PRIMITIVE_TICK`(0.6)
- [ ] **物理化の視覚合図**: 確定と同時に線がわずかに落下・たわむ（ばね付きジョイント物理が演出を無料で提供 — §7.2 方式C の副次効果）

**推奨（差がつく層）**
- [ ] ペン先ダスト: 2〜5個/フレーム（速度比例）、寿命 0.2〜0.5s、サイズ 2〜6px、線色→透明フェード
- [ ] インク消費量 = 星評価の連動を描画中に予告表示（「この線だと★2」）
- [ ] 描画中 continuous ハプティクス（iOS intensity 0.2〜0.35 / Android LOW_TICK 30〜60ms 間隔）— 任意

### 4.2 場面2: 車が走り出す瞬間（anticipation → release）

**必須**
- [ ] **溜め（0.3〜0.5s 固定・スキップ不可の短い儀式）**: エンジンレブ音ピッチ 1.0→1.4 上昇 + 車体後傾 squash（後方5〜8°、縦0.92/横1.08、0.2s ease-in）+ 車輪空転・後方に煙
- [ ] **解放**: ダストパーティクル 10〜20個一斉放出 + 車体前方 stretch（横1.15/縦0.9→100msで復帰, ease-out）+ 低域を効かせた発進バースト音（Vlambeer「bassを足せ」）+ ハプティクス iOS `.medium` / Android `PRIMITIVE_THUD`(0.8)
- [ ] **カメラ**: lerp 追従（係数 0.08〜0.15 @60fps）+ look-ahead（進行方向へ車体1〜2台分、速度比例）。発進時カメラキック（逆方向 8〜16px → 0.3s 復帰）
- [ ] **車輪回転・サスバウンス**: 実速度同期の車輪回転 + wheel joint サスの上下動（Hill Climb Racing の「気持ちよく誇張された物理」基準）
- [ ] **エンジン音の速度→ピッチ変調**: 1.0→1.5 連続 + 0.25 刻みのギアチェンジ風段付き
- [ ] **橋の軋みフィードバック**（本作固有の差別化 juice）: ジョイント応力 stress 0.6〜1.0 帯域を (a) 軋みSFX音量/ピッチ (b) 線分色 白→黄→赤 (c) 微振動・粉パーティクル (d) ハプティクス弱連打 にマッピング（Poly Bridge 型応力可視化。物理値から無料で手に入る）
- [ ] **コイン取得音の半音上昇**: 連続取得ごとに +1 semitone（×1.0595）、上限+12、**1〜1.5s 途切れでリセット**。取得ポップ（scale 1.0→1.3→0, 150ms）+ キラ粒子 4〜8個
- [ ] 破断時: クラック音 + 破片 + カメラ trauma +=0.5（§4.4）+ 折れ口ハイライト

**推奨**
- [ ] トラウマ方式 screen shake: `shake = trauma²`、Perlin ノイズ、maxOffset 16〜30px / maxAngle 5〜10°、freq 15〜25Hz。加算目安: 発進0.15 / 着地0.2〜0.3 / クラッシュ0.5 / ゴール0.4
- [ ] 速度連動ズームアウト +10〜20%、スピード線（最高速60%超で出現）
- [ ] スキッドマーク・描線の permanence（痕跡を残す）
- [ ] 着地ハプティクス `.heavy` / `PRIMITIVE_THUD`(1.0)（大ジャンプ後のみ）
- [ ] コインのアーチ状配置（0.1〜0.2s 間隔で取れるリズム配置）+ コイン取得ハプティクスは2〜3枚に1回に間引く

### 4.3 場面3: ゴール（5拍構成: 時間演出→爆発→評価→報酬→即・次へ。全体3〜4秒・スキップ可）

**必須**
- [ ] **hit-stop**: 旗接触フレームで 80〜120ms の完全停止（推奨→必須に格上げ。ゴールの「切断面」を作る）
- [ ] **スローモーション**: 直後に timeScale 0.3 へ即時変更、実時間 0.3〜0.5s 維持 → 0.2〜0.3s で 1.0 へ lerp 復帰。物理の fixedDeltaTime も連動（カクつき防止）。スロー中カメラ 15〜25% ズームイン
- [ ] **Confetti 2段構成**: ①ゴール地点から左右2門キャノン（各40〜60個、斜め上45〜70°、拡散30°）②0.3s 遅れて上部から降下レイン（60〜100個、2〜3s）。紙片は回転±720°/s + sin 揺れ、重力0.2〜0.4倍。発射音「ポンッ」×2（左右50msずらし）+ `.heavy` ハプティクス
- [ ] **星の順次出現**: 200〜300ms 間隔、各星 scale 0→1.3→1.0（250ms, ease-out-back）+ 衝撃波リング。音は上昇アルペジオ（ド・ミ・ソ）、3つ目だけ豪華（シンバル追加）。ハプティクス漸増 light→medium→heavy
- [ ] **報酬カウントアップ**: 0→獲得額を 0.8〜1.5s（ease-out）。チック音 30〜60ms 間隔でピッチ 1.0→1.3 上昇。**タップで即スキップ**
- [ ] コインバースト→回収: 10〜30枚放射爆発 → 各 20〜40ms ずらしでカウンターへ飛行（各0.4〜0.6s, ease-in）。到達ごとに半音上昇チン音 + カウンター scale パンチ(1.0→1.2→1.0, 100ms)
- [ ] **Next ボタン**: 演出完了後 1.5〜2.5s で活性化、脈動誘導（scale ±5%、周期0.8s）
- [ ] BGM ダッキング: ゴール瞬間 -6〜-9dB（0.2s）で SFX を立てる

**横断必須**
- [ ] ハプティクスの端末対応チェック（Android `areAllPrimitivesSupported()` 必須 — 未対応が1つでもあると composition 全体が無音）+ フォールバック + 設定OFF提供
- [ ] hit-stop は1レベル1〜2回まで（ゴール・大クラッシュ限定）。「頻繁なアクションほど juice はシンプルに」

## 5. メタ進行決定

### 5.1 採用パワーアップ（5種 + 番外1、証拠付き）

| 優先 | パワーアップ | 体験が変わる理由 | 証拠 | 導入時期 |
|---|---|---|---|---|
| 1 | **新車両アンロック**（重量・車輪数・サイズ・特性が異なる） | 同じレベルでも描くべき橋が変わる = 横方向進行の本命 | JJ Vehicles / HCR2 固有枠（挙動が変わるアンロックの実証）、Crossy Road 400+収集で3ヶ月$10M、Bridge Race キャラ80+ | v1.1: 観賞スキン10〜20種ガチャ → v1.2: 物理特性付き |

## 6. 収益化方針

### 6.1 大方針: 無料DL → 後日広告導入。ただし**広告枠は v1.0 の設計に織り込む**

- v1.0（ローカルMVP〜初期リリース）は広告SDKを同梱しない（Glowgrid 方式の「Data Not Collected」で審査を軽くする選択肢も残る）。
- ただし**過去2作の最大の欠落は「収益化実装ゼロ」**（00 §f4）。同じ轍を踏まないため、v1.0 から以下を必須とする:
  1. **`AdInterface` 抽象**（`showRewarded(placement)` / `showInterstitial(placement)` / `isReady()` / イベントコールバック）を定義し、v1.0 は `NoopAdProvider`（常に即時成功 or 非表示）を注入。UI 側の広告導線（RVボタン等）はフラグで非表示化できる形で実装まで済ませる。
  2. 広告プレースメント ID・表示タイミング・頻度キャップを **Remote Config 化前提の定数**として `TuningConstants` に定義。
  3. Poki / CrazyGames SDK のイベント（`gameplayStart/Stop`, `commercialBreak`, `rewardedBreak`）も同じ抽象の別実装として設計（Web 配信時に差し替えるだけにする）。

### 6.2 広告フォーマットと表示タイミング（導入時の確定仕様）

| フォーマット | プレースメント | 頻度・条件 | 根拠 |

| **リワード動画（主役）** | ①クリア画面「コイン×3〜5で受け取る」常設 ②失敗時コンティニュー（その場から再開ではなく「このレベルのヒント/スキップ」型）③車両1日試乗 ④安価UG無料化（序盤5Lv分） | 1セッション6〜10回提示、日次キャップ15〜20回。**opt-in のみ** | watch rate 40〜70%、完走90%+、RVはリテンション中立〜プラス。eCPM $16〜20 でインタースティシャルより高単価 |

## 7. 技術スタック決定

### 7.1 スタック確定（05 の推奨を 06_gap_1 / 06_gap_3 の追加検証込みで採択）

```
言語:         TypeScript
エンジン:      Phaser 4.1.x（新WebGLレンダラ）
物理:         Phaser Box2D（Box2D v3 JSポート / MIT / 65KB）
ビルド:       Vite（dev: HMR でブラウザ即プレイ / build: 静的バンドル）
ネイティブ化:  Capacitor 8（※7ではなく8。プラグイン生態系が v8 世代に移行済み）
広告/IAP/計測: §6.3 のとおり（v1.0 はインターフェースのみ）
ハプティクス:  @capacitor/haptics（+ 将来 CoreHaptics/VibrationEffect 拡張プラグイン）
サウンド:      Phaser Sound（Web Audio）
```

**採択理由**: ①Box2D v3 の chain / capsule / wheel joint / **ジョイント破断API** を TS ネイティブで直接叩ける唯一の構成 ②`npm run dev` でブラウザ即プレイ + HMR — 物理チューニング（本作の工数の大半）の反復速度が品質に直結 ③Web 版がそのまま Poki/CrazyGames 配布・需要検証資産になる ④LiveOps 6領域（IAP/計測/RC/A/B/クラッシュ/プライバシー）すべて「不可能」判定ゼロを確認済み（06_gap_3）。商用前例: Vampire Survivors モバイル版（Capacitor 製）。

**不採用理由の要点**: Unity（TS不可・WebGLビルド30分超でブラウザ反復不能）、Godot（TS不可・2D wheel joint 非搭載・Web 40MB）、Cocos（次点 8.1。性能スパイク失敗時の移植先として保持 — ロジック層は TS のまま再利用可能）。

### 7.2 物理実装パターン確定

**描線 → ボディ化: 「セグメント化カプセルチェーン + ばね付きジョイント + 破断」を第一方式とする**（06_gap_1 の結論。05 の単一剛体 compound は**フォールバック層に格下げ**）:

```
[入力層]  pointer点列 → RDP間引き → 等間隔リサンプル（セグメント長 0.5〜0.8m 相当）
[物理層]  カプセルセグメント×N（N=8〜24、上限32）を revolute joint で連結
          - enableSpring: hertz 4〜8（曲げ剛性=たわみ）, dampingRatio 0.6〜0.8
          - enableLimit: ±0.2〜0.4 rad（最大折れ角）
          - collideConnected=false, 同一ストローク内は filter.groupIndex=-strokeId で自己衝突オフ
          - 破断: raw = |F|/breakForce + |τ|/breakTorque を EMA平滑（0.85/0.15）し
            stress > 1.0 で b2DestroyJoint（部分崩落）。stress 0.6〜1.0 は軋み演出へ直結
          - 破断後の孤立チェーン片は数秒後に車と非衝突化 + フェードアウト
[描画層]  物理セグメント位置から Catmull-Rom スプラインで滑らかな1本線を再構成
          （物理Nと描画頂点を分離。破断点で描画パス分割 + 折れ口ギザギザ）
[保険層]  同じ入力/描画層のまま物理層だけ単一 compound（旧05方式）に差し替え可能にする
          （低性能端末向け設定 or スパイク失敗時のフォールバック）
```

- 根拠: Box2D 公式 Driving サンプル（カプセル20枚 revolute 橋を車が渡る = 本作コアループと同型）と Bridge サンプル（160枚, hertz 2.0/damping 0.7）が実在実証。破断は公式 BreakableJoint サンプルが雛形で、Phaser Box2D が必要API（`b2Joint_GetConstraintForce/Torque`, `b2DestroyJoint`）をエクスポート済みであることをソース確認済み。
- 安定性の実務指針: 線の総質量を車の 0.3〜1.0 倍、セグメント単体との質量比 ≤ ~30:1、subStep 4→6〜8、N は 10〜16 から開始。

**地形**: static body の **chain shape** 一択（片側衝突・ゴースト衝突なし）。

**車**: **chassis（角丸ボックス or カプセル）1 + wheel（円）2 + b2WheelJoint ×2** の標準構成。サス初期値 `hertz ≈ 4Hz / dampingRatio ≈ 0.7`、駆動は後輪 `enableMotor + motorSpeed + maxMotorTorque`、タイヤ摩擦 0.8〜1.2 / 地形・描線 0.6〜0.9 / restitution 0。

**時間**: 固定タイムステップ `dt = 1/60` + アキュムレータ + 補間レンダリング（Phaser Box2D の WorldStep は accumulator 内蔵。補間は必要に応じ自前追加）。120Hz 端末はレンダのみ高リフレッシュ、物理は 60Hz 固定。

**決定論の扱い（00 の最重要技術判断への回答）**: JSポートは `Math.sin/cos/atan2` 依存のため**ブラウザ間ビット一致は保証されない**。よって決定論契約は既存2作の「完全決定論」から次のように緩和して確定する:
- CI（固定 Node = 固定 V8）内: **ビット一致 / 終了状態ハッシュ一致**で回帰検出（Box2D 本家CI方式）
- 実機・エンジン更新時: 許容帯検証（成功/失敗一致 + 最終車両位置 ε=0.05m + tick ±30 を仮置き、感度分析で校正）
- リプレイ/ゴースト共有機能は「入力再生」ではなく「位置サンプル再生」で実装
- ビット決定論が将来必須になったら (a) ポートの三角関数差し替えフォーク (b) `rapier2d-deterministic`（WASM）移行の2択を再評価

**環境非依存のコンポーネント設計**（前提4の具体化）:
- `Engine`（物理・ルール。**Phaser 非依存・ヘッドレス実行可能** — CI ボット検証の前提）/ `Render`（Phaser。エンジン出力の観測者、書き戻し禁止）/ `Meta`（経済・永続化）/ `Platform`（Ad / Haptics / Analytics / IAP / Storage の各インターフェース + Capacitor 実装 + Web 実装 + Noop 実装）/ `TuningConstants` + レベルJSON に数値全集約。

### 7.3 着手週の必須スパイク（実装開始のゲート）

1. **描線物理方式スパイク**（06_gap_1 §6.5）: 同一シーン（崖8m + 固定描線 + 車）で A:単一compound / B:自由ヒンジ / C:ばね付きヒンジ / D:weld 鎖 × N=8/16/24/32 を切替比較。中級 Android 実機（Snapdragon 6xx / Helio G 系）の Capacitor WebView で計測。**合格基準: p95 step ≤ 4ms かつ 60fps、C or D で「荷重たわみが視認でき破断が誇張なく決まる」**。不合格時のみ A + 描画層のみの演出たわみへ後退。
2. **カプセル×車輪の接触品質検証**: Phaser Box2D に未マージ修正PR（#24 カプセル衝突 manifold）があるため最初に検証。問題があれば PR 取り込みフォーク or セグメント形状を rounded box に変更。
3. **run-to-run 決定論実測**: 同一レベル×同一描線×1000回実行でハッシュ比較（CI ゲートの前提確認）。

## 9. リスクと対策

| # | リスク | 影響 | 確度 | 対策 |
|---|---|---|---|---|
| 1 | **中級 Android WebView で 60fps 割れ** | 爽快感の根幹毀損 | 中 | 着手週スパイク必須（§7.3）。NG時: resolution scale 0.75 / パーティクル削減 / 物理フォールバックA。最終手段は Cocos Creator 移植（TS ロジック再利用可） |
| 2 | **Phaser Box2D のメンテ停滞**（最終push 2025-04、カプセル衝突の未マージPR） | 接触品質バグ・将来の追随不能 | 高（事実）〜中（実害） | MIT フォーク前提。スパイク1でカプセル×車輪を最初に検証。物理層 thin wrapper で Rapier2D への移行路を確保 |
| 3 | **ばね鎖チューニング沼**（硬いと崩落が地味、柔らかいと暴れる） | 開発期間肥大・手触り不良 | 中〜高 | 全パラメータをデバッグスライダ化して実機で回す（BreakableJoint サンプル運用）。破断閾値は「車静止荷重の2〜3倍」から。物理Nを増やさず描画スプラインで見た目を稼ぐ |
| 4 | **コンテンツ量産が市場基準（500面）に届かない** | ソフトローンチ基準未達・D30 低迷 | 中 | テンプレート×パラメータ変奏×ボット検証の3層パイプライン + インゲームエディタに初期投資（1〜2週）。目標: ツール整備後 週15〜20面/人。ローンチ段階要件はフェーズ分割（MVP 15面 → SL 300+） |
| 5 | **サブジャンル自体が下り坂**（新規ヒットは2022年が最後） | UA スケール不能 | 中 | UA前提にしない。Web（Poki Web Fit Test = 無料1万人検証）→ 少額CPIテスト（$360〜、CPI<$0.25/D1≥35% でパブ持ち込み）→ 各段階に kill 基準（06_gap_4 §5）。差別化は品質と共有適性に置く |
| 6 | **広告導入時の体験自壊**（競合の轍） | レビュー炎上・D1急落 | 中 | §6.2 の頻度キャップを Remote Config 化して遠隔調整可能に。インタースティシャルは Level 8 まで封印・失敗直後に出さない・広告除去IAPを必ず併設 |
| 7 | **ブラウザ/端末間の物理非決定論** | リプレイ共有・検証の不一致 | 高（事実）/ 低（MVP実害） | 決定論契約を「CI内ビット一致 + 実機許容帯」に再定義済み（§7.2）。ゴースト共有は位置サンプル再生。ビット決定論が必須化したら rapier2d-deterministic へ |
| 8 | **Google Play 新規個人アカウントの「12テスター×14日」要件** | CPIテスト・リリースの機動力喪失 | 高（事実） | プロジェクト初週にストアアカウント問題を解決（組織アカウント取得 or 既存アカウント利用 or テスト前倒し）。GTM フェーズ0 のタスクに固定 |
| 9 | **収益化・計測の実装が過去2作同様に先送りされ続ける** | 商用レベル未達（前提1違反） | 中 | v1.0 から AdInterface/Analytics 抽象を必須スコープに入れた（§6.1・§8.1）。brand/store 工程（Glowgrid RELEASE.md 雛形流用 + Play Console 版新規作成）を計画に明記 |
| 10 | **juice 過剰でテンポ死** | ループ1周が延びて離脱 | 低〜中 | 全演出スキップ可・失敗時は演出最軽量・ループ1周≤40秒を数値契約テスト化（00 のテンポ契約方式） |
