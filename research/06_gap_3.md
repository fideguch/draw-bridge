## 0. 前提の更新: Capacitor 7 は保守終了目前、Capacitor 8 を採用すべき

調査指示は「Capacitor 7」前提だったが、**Capacitor 8 が 2025年11–12月にリリース済み**であり、Capacitor の公式サポートポリシーは「メジャーバージョンは次版リリース後 6ヶ月の保守（＋エンタープライズ契約で 6ヶ月延長）」。つまり **2026年7月現在、Capacitor 7 は標準保守期間の終端にある**。（確度: 高）

### 1.2 判定と推奨

- **判定: 問題なく可能。** 広告除去（非消耗型）＋コインパック（消耗型）というハイブリッドカジュアルの標準構成は、RevenueCat 公式 Capacitor プラグインで完全にカバーされる。iOS の In-App Purchase capability 有効化、Android の `launchMode: singleTop` 設定などの定型作業のみ。
- **推奨: RevenueCat を第一候補**。理由: (1) 公式サポート＋高頻度リリースで StoreKit 2 / PBL 8 追随が速い、(2) レシート検証・返金検知・収益ダッシュボードが内蔵で、個人開発のハイブリッドカジュアル（初期は MTR $2,500 未満 = 無料）に最適、(3) 後述の A/B テスト（価格実験）や LTV 計測との統合が容易。
- 想定リスク: 収益が伸びた場合の 1% 課金（グロス基準）。広告除去 $4.99 単発中心なら MTR $2,500/月 = 約500本/月 までは無料であり、初期リスクはゼロに近い。
- cordova-plugin-purchase は「手数料ゼロで自己完結したい」場合の代替。ただしレシート検証を自前実装する工数と、個人メンテ由来の追随リスク（PBL 8 対応でリグレッション報告あり）を織り込むこと（確度: 中）。

## 2. アナリティクス: Firebase Analytics + ゲーム KPI — 判定「問題なく可能」（確度: 高）

### 2.1 `@capacitor-firebase/analytics`（capawesome）の実用性

- capawesome チームの Firebase プラグイン群は **12 プラグイン構成（Analytics / Crashlytics / Remote Config / Performance ほか）で、Capacitor Firebase 統合のデファクト**。GitHub 522★・332 リリース・2026年6月にもリリースあり、活発に保守されている（確度: 高）。
- 重要な設計ポイント: このプラグインは WebView 内の Firebase **JS SDK ではなくネイティブ SDK（iOS/Android）を叩く**。したがって GA4/Firebase の自動イベント（`first_open`, `session_start`, `app_update` 等）、オーディエンス、A/B テストのターゲティングがネイティブアプリと同等に機能する。WebView 内 JS SDK 直接利用で起こる cookie / measurement-protocol 問題を回避できる。
- `logEvent(name, params)` / `setUserId` / `setUserProperty` / `setCurrentScreen` を提供。GA4 の制約（イベント名 500 種まで、1イベント25パラメータ等）は通常のネイティブアプリと同一。

### 2.2 ゲーム KPI に必要なイベント設計（このスタックでそのまま実装可能）

| KPI | 実装 | 備考 |
|-----|------|------|
| D1/D7 リテンション | **追加実装不要**。`first_open` + セッション自動計測で Firebase/GA4 のリテンションコホートが自動生成 | GA4 探索レポートのコホート分析で D1/D7 を厳密に定義可能 |
| レベルファネル | GA4 ゲーム推奨イベント `level_start` / `level_end`（`success` param）/ `level_up` を `logEvent` で送信 | ゲーム向けレポートテンプレートはこれらのイベント名を前提とする |
| ソフト通貨経済 | `earn_virtual_currency` / `spend_virtual_currency`（`virtual_currency_name`, `value`, `item_name`） | メタ経済（コイン獲得/消費バランス）の監視に必須。A/B 分析の指標にもなる |
| IAP 収益 | RevenueCat → Firebase/GA4 連携 or `purchase` イベント。ネイティブ課金は GA4 が自動収集 | |
| 広告収益（ユーザー単位） | 後述 2.3 | |
| 進行度ファネル自由定義 | `design` 相当のカスタムイベント（例: `tutorial_step`, `upgrade_purchased`） | 25 パラメータ/イベントの範囲で自由 |

### 2.3 広告収益計測の落とし穴と回避策（確度: 高）

- `@capacitor-community/admob` は **Impression-Level Ad Revenue（`onPaidEvent`）を JS へ公開していない**（Issue #414 / #293 で機能要望のまま未実装）。プラグイン単体ではユーザー単位の広告 LTV が取れない。
- **回避策: AdMob アプリを Firebase にリンクする**。リンクすると Google Mobile Ads SDK が **`ad_impression` イベント（platform / source / currency / value 付き）を自動で GA4 に記録**し、ARPU / LTV に広告収益が自動合算される。**コード追加ゼロ**で成立するため、Capacitor プラグインの制約を完全に迂回できる。
- 精緻な ILAR（メディエーション込み impression 単位収益を MMP に流す等）が必要になるのは UA を回す段階であり、本プロジェクトのオーガニック中心戦略では AdMob⇔Firebase リンクで十分。

### 2.4 GameAnalytics 等ゲーム特化 SDK の Capacitor 対応

- **公式の Capacitor/Cordova SDK は存在しない**（確度: 高）。
- ただし GameAnalytics には**公式 JavaScript SDK（npm `gameanalytics`、HTML5 ゲーム向け）**があり、「navigator オブジェクトが存在するブラウザ環境」を前提とする＝**Capacitor の WebView 内でそのまま動作する設計**。business / resource / progression / design / ads / error の6イベント体系＋Remote Configs / A/B テスト機能を含む（確度: 中。WebView 内動作は設計上動くはずだが、公式が Capacitor を明示サポートしているわけではない）。
- 懸念: npm 最終公開が約2年前（v4.4.7）で更新頻度が低い。**主計測基盤は Firebase とし、GameAnalytics はゲーム KPI ベンチマーク（無料）目的のセカンダリ計測として任意追加**が妥当。
- 判定: **Firebase Analytics = 問題なく可能 / GameAnalytics = 回避策あり（JS SDK を WebView で使用）**。

## 3. Remote Config / A/B テスト — 判定「問題なく可能」（確度: 高）

### 3.1 `@capacitor-firebase/remote-config` の成熟度

- capawesome Firebase プラグイン群の一部として保守（v8.x = active、v7.x = Capacitor 7 用）。
- API カバレッジは実運用に十分:
  - `fetchAndActivate()` / `fetchConfig()` / `activate()`
  - `getBoolean()` / `getNumber()` / `getString()` / `getAll()`
  - `setSettings()`（fetch timeout / **minimumFetchInterval** 設定）
  - **`addConfigUpdateListener()` — リアルタイム設定更新（iOS/Android 対応）**。公開中バージョンへの即時パラメータ反映（緊急ネジ回し）が可能
- 条件ターゲティング（ユーザープロパティ / オーディエンス）には Analytics プラグイン併用が必要（前節で導入済みのため問題なし）。
- iOS では SPM 利用時に package identity 衝突回避の設定が1点必要（ドキュメント記載あり、既知の定型作業）。

### 3.2 Firebase A/B Testing はこのスタックで機能するか

- **機能する（確度: 高）**。Firebase A/B Testing は Remote Config ＋ Google Analytics の**ネイティブ SDK 上**で動作する仕組みであり、capawesome プラグインはまさにネイティブ SDK のラッパーであるため、実験群の割当・アクティベーションイベント・目標指標（retention / revenue / カスタムイベント）の統計評価がそのまま使える。
- Firebase A/B Testing の廃止予定は確認されず、むしろ **2026年3月に Web 対応が追加される等、投資が継続**している（Firebase ML など他サービスの廃止は発表されているが A/B Testing は対象外）。
- 戦略前提だった「remote config による自前 A/B」も可能: Remote Config のパーセンタイル条件（`percent` 条件）で自前分割 → GA4 でユーザープロパティ別に集計、という Firebase A/B Testing を使わない運用も同一プラグインで実装できる。

### 3.3 経済パラメータ A/B の運用パターン（ハイブリッドカジュアル標準形）

1. **パラメータ設計**: `coin_reward_per_level`, `upgrade_cost_curve`, `interstitial_interval_sec`, `rewarded_coin_multiplier`, `remove_ads_price_tier` 等を Remote Config キーとして定義し、Phaser 側は必ずデフォルト値を同梱（オフライン初回起動対策）。
2. **取得戦略**: 起動時 `fetchAndActivate()`、ただし**セッション途中の経済パラメータ変更はプレイ体験を壊すため、fetch は起動時・activate は次セッション**（loading 画面）に分離するのが Firebase 公式推奨のロード戦略。
3. **実験設計**: Firebase A/B Testing で「目標=D7 リテンション / 広告収益 / `spend_virtual_currency` 量」を設定し、`level_end(success)` ファネルと併読。
4. **緊急運用**: `addConfigUpdateListener` で難易度・広告頻度の緊急調整（審査再提出なし）。
- 判定: **問題なく可能**。Unity の Remote Config / Firebase 構成と実質同等の運用ができる。

## 4. クラッシュ・性能計測 — 判定「問題なく可能」（構成に注意、確度: 高）

Capacitor ゲームの障害は2層で起こる: **(a) WebView 内 JS エラー**（Phaser のゲームロジック起因、アプリは落ちない）と **(b) ネイティブクラッシュ**（プラグイン・OS・OOM 起因）。両取りの経路は2つある。

### 4.1 経路A: Sentry（`@sentry/capacitor`）— 一発で両取り

- **公式 SDK**。v2.0 で Capacitor 7 対応、v3.x で Capacitor 8 対応（Sentry JS v10 ベース）。
- **JS エラー（unhandled exception / promise rejection）とネイティブクラッシュ（iOS/Android hard crash）の双方を単一 SDK で捕捉**。Release Health（crash-free rate）・パフォーマンス計測も対応。Android 16KB ページサイズ対応も v1.5.0+ で済み。
- 既知の弱点: iOS の JS エラーにネイティブメタデータが付かないケースが Issue 報告されている（#138）。致命的ではない。
- 無料枠（Developer プラン）あり。個人開発初期は無料で回る。

### 4.2 経路B: Crashlytics（`@capacitor-firebase/crashlytics`）＋ JS エラーの手動転送

- ネイティブクラッシュは Crashlytics が標準どおり捕捉。無料・Firebase コンソールに統合される点が利点。
- WebView 内 JS エラーは自動では乗らない。回避策: `window.onerror` / `unhandledrejection` をフックし、**`recordException({ message, stacktrace })` に stacktrace.js で整形したスタックを渡して非致命エラーとして送る**（プラグインが stacktrace パラメータを公式サポート）。
- 制約: (1) JS エラーは常に「non-fatal」扱いで crash-free rate に反映されない、(2) iOS では domain/code 指定時に stacktrace が無視される仕様あり、(3) Android の WebView レンダラプロセスクラッシュ（`onRenderProcessGone`）は自動捕捉されない（capawesome Issue #100 で議論、未解決）。

### 4.3 推奨

- **ゲームロジックの品質監視が主目的なら Sentry 一本が最小工数で両取り**でき、リリースヘルスも見える。Firebase エコシステムに寄せたい（コンソール一枚に統一したい）なら Crashlytics ＋ window.onerror ブリッジ。両者併用も技術的には可能。
- 性能計測: `@capacitor-firebase/performance` プラグインも存在（起動時間・ネットワーク）。ゲームの FPS 計測は Phaser 側（`game.loop.actualFps`）からカスタムイベントで送るのが現実的。
- 判定: **問題なく可能**。「WebView 内 JS エラー＋ネイティブクラッシュの両取り」は Sentry なら標準機能、Crashlytics でも確立された回避策で実現できる。

## 5. 広告メディエーション続報 — 判定「回避策あり」（確度: 中）

### 5.1 AppLovin MAX × Capacitor の現状

- AppLovin 公式の Capacitor 対応要望（AppLovin-MAX-Cordova Issue #32）は **2023年2月に「not planned」でクローズ**。公式サポートは今も無い（確度: 高）。
- ただし MAX Cordova プラグイン自体は保守継続中（最新 **v2.1.0、Android/iOS SDK 13.2.0 同梱**。v2.0.0 → v2.1.0 と 2026 年もリリースが出ている）。Capacitor は Cordova プラグイン互換層を持つため、`npm i cordova-plugin-applovin-max` で導入する**コミュニティ実績（ionic-applovin-integration 等）は存在する**。
- リスク評価: 動作報告はあるが「AppLovin が Capacitor 上の不具合をサポートしない」「Capacitor メジャーアップデート時に互換層起因の破損リスク」という運用リスクが残る。Capacitor 7/8 上での大規模商用実績は確認できなかった（確度: 低〜中）。**MAX を収益化のクリティカルパスに置くべきではない**。

### 5.2 AdMob 単独で始めた場合の機会損失の定量感

- 業界ベンチマーク（複数のメディエーション/収益化ベンダー公表値）:
  - リワード動画 eCPM（Tier1: US/UK/DE/JP）: **単独ネットワークで $15–30**、メディエーション込みで $18–45 程度
  - メディエーション導入による収益改善: **+40〜100%（メディエーションベンダー公称）〜 +40–60%（比較サイト）**。ただしこれらは**メディエーションを売る側のマーケティング数値**であり、小規模タイトル・オーガニック中心では改善幅はより小さいのが通説（確度: 低〜中）
  - AdMob 単独のリワード広告フィル率: **主要地域で 90–95%+**（Tier1 ではフィル率自体はほぼ問題にならない。フィルが崩れるのは Tier3 やフロア価格設定ミス時）（確度: 中）
- 解釈: **DAU 数千規模の立ち上げ期において、AdMob 単独で始めることによる機会損失は「収益の2〜4割」レンジと見るのが妥当だが、絶対額が小さい期間は運用簡素化の価値が勝る**。ハイブリッドカジュアル戦略（IAP 広告除去 $4.99 が柱、広告はリワード中心）なら、広告 eCPM 最大化の優先度はピュア広告型より一段低い。

### 5.3 現実的なメディエーション経路（このスタック固有）

1. **フェーズ1（ローンチ〜PMF）**: `@capacitor-community/admob` v7/v8 で AdMob 単独（bidding 有効）。UMP/ATT/SSV（リワードのサーバーサイド検証）内蔵で、リワード・インタースティシャル・バナー・App Open まで揃う。
2. **フェーズ2（規模化）**: **AdMob mediation を「ネイティブ層だけで」追加**。AdMob のメディエーションはアダプタ（例: `com.google.ads.mediation:ironsource`, Unity Ads, Mintegral 等）を `android/app/build.gradle` と iOS の依存管理に追加し、AdMob 管理画面でウォーターフォール/bidding を構成すれば、**JS プラグイン API は変更不要**で複数デマンドが競合する（広告のロード/表示 API は AdMob のまま）。Capacitor でもネイティブ依存の追加は標準手順であり、プラグイン改修なしで成立する（確度: 中。アダプタ方式の設計上は成立するが、Capacitor での網羅的な動作実績報告は少ない）。
3. **フェーズ3（オプション）**: それでも足りなければ MAX Cordova プラグインを互換層で検証導入。ここは「実験」扱い。
- 判定: **回避策あり**。AdMob 単独開始 → AdMob mediation 拡張が本命経路。MAX 非対応はスタックの弱点だが、致命傷ではない。

## 6. プライバシー要件（ATT / SKAdNetwork / UMP / PrivacyInfo.xcprivacy）— 判定「問題なく可能」（確度: 高）

| 要件 | 実装経路 | 状態 |
|------|---------|------|
| **ATT（App Tracking Transparency）** | ① `@capacitor-community/admob` 内蔵の `trackingAuthorizationStatus()` / `requestTrackingAuthorization()`、または ② 専用プラグイン `capacitor-plugin-app-tracking-transparency`（v3.x が SPM 対応で Capacitor 7/8 推奨、v2.x は CocoaPods 用）。`NSUserTrackingUsageDescription` を Info.plist に追加 | 問題なし |
| **UMP（Google User Messaging Platform / GDPR 同意）** | `@capacitor-community/admob` に内蔵: `requestConsentInfo()` → `showConsentForm()`、`showPrivacyOptionsForm()`（プライバシー設定の再表示）まで対応。AdMob 管理画面で GDPR/IDFA メッセージを作成して使う標準フロー | 問題なし |
| **SKAdNetwork（SKAN 4 / AdAttributionKit）** | Info.plist に `SKAdNetworkItems`（Google の `cstr6suwn9.skadnetwork` ＋各ネットワークの ID）を列挙。SKAN の登録・postback 処理自体は Google Mobile Ads SDK がネイティブで実施するため、Capacitor 側の追加実装は不要。オーガニック中心戦略なら実装は Info.plist 追記のみで足りる | 問題なし |
| **PrivacyInfo.xcprivacy（Privacy Manifest）** | Capacitor 本体（>=5.7.4 / 6+）は SDK 側の Privacy Manifest を同梱済み。アプリ側は `ios/App/Resources/PrivacyInfo.xcprivacy` を追加（Ionic VS Code 拡張が使用プラグインから required-reason API を検出して自動生成可能）。Firebase / Google Mobile Ads / RevenueCat 等の主要 SDK は各自のマニフェストを同梱している | 問題なし |

- 判定: **問題なく可能**。いずれも確立された定型手順で、Unity 等と比べて特別な劣位はない。

## 総合判定: エンジン再考は不要

### 判定根拠

1. **6領域すべてで「不可能」判定はゼロ**。IAP・Analytics・Remote Config/A/B・クラッシュ計測の4本柱は、いずれも「公式 or デファクトのアクティブに保守された Capacitor プラグイン」で成立し、Unity で構築する場合と機能面の遜色がない。
2. 戦略前提だった **「広告除去 IAP $4.99」「remote config による自前 A/B での経済チューニング」は、RevenueCat（MTR $2,500 まで無料）＋ capawesome Remote Config（リアルタイム更新付き）で完全に実現可能**。むしろ Firebase A/B Testing の統計評価が使えるぶん「自前 A/B」前提より強い。
3. 唯一の構造的弱点は **AppLovin MAX 公式非対応**（メディエーションの選択肢が狭い）だが、(a) ハイブリッドカジュアル戦略では広告は収益の従で IAP が主、(b) AdMob mediation をネイティブ層で足す回避策がある、ことから **エンジン評価を覆す水準ではない**。Phaser+Capacitor 8.6/10 の評価は維持、あるいは「LiveOps 基盤の確認済み」でむしろ確度が上がった。
4. **条件: Capacitor 7 ではなく Capacitor 8 で開始すること**。プラグイン各社の active support は既に v8 世代（capawesome v8 / admob v8 / RevenueCat v13 / Sentry v3）に移っており、Capacitor 7 縛りだと「deprecated 版プラグインの寄せ集め」になる。

- capawesome プラグインの「active support は最新 Capacitor のみ」ポリシー → Capacitor のメジャーアップに年1回追随する運用前提を持つこと。
